## 1-1. XML とは何か

**XML（eXtensible Markup Language）** は、データを「意味のある構造」で表現するためのマークアップ言語。

### HTML との比較

| 観点 | HTML | XML |
|------|------|-----|
| 目的 | データを**表示**する | データを**表現・転送**する |
| タグ | 決められたタグのみ | 自分でタグを定義できる |
| 厳密さ | 比較的ゆるい | 厳密な構文ルールがある |

### JSON との比較

| 差別化点 | 説明 |
|---------|------|
| **スキーマ・バリデーション機能（DTD）** | データの「正しさ」を定義・検証できる ⚠️ XXEの舞台 |
| **名前空間（Namespace）** | 異なるシステムのタグが混在しても衝突しない |
| **コメント・処理命令** | `<!-- -->` コメントや `<?...?>` 処理命令を持てる |
| **歴史的な普及経緯** | SOAPなど企業間連携プロトコルはXMLベースで設計された |

> **ポイント**：JSON でもキーと値でメタ情報は表現できる。属性値の有無は本質的な差別化点ではない。

---

## 1-2. XML の基本構造

```xml
<?xml version="1.0" encoding="UTF-8"?>   ← XML宣言
<user>                                     ← ルート要素（開始タグ）
  <name>Taro</name>                        ← 子要素
  <age>30</age>
  <email>taro@example.com</email>
</user>                                    ← ルート要素（終了タグ）
```

---

## 1-3. 構成要素の詳細

### 1-3-1. XML 宣言

```xml
<?xml version="1.0" encoding="UTF-8"?>
```

| 部分 | 意味 |
|------|------|
| `version="1.0"` | XML のバージョン指定 |
| `encoding="UTF-8"` | 文字エンコーディングの指定 |

- 必須ではないが、先頭に書くのが慣例
- **パーサーはまずここを読んで、どう解釈するか決める**

---

### 1-3-2. 要素（Element）

```xml
<name>Taro</name>
```

- `<name>` が**開始タグ**、`</name>` が**終了タグ**
- タグで囲まれた `Taro` が**コンテンツ（内容）**
- 開始タグと終了タグは**必ずペア**（HTML より厳密）

#### 入れ子構造（ネスト）

```xml
<user>
  <profile>
    <name>Taro</name>   ← profileの子、userの孫
  </profile>
</user>
```

- 必ず**1つのルート要素**がすべてを包む
- 要素は何層でも入れ子にできる

---

### 1-3-3. 属性（Attribute）

```xml
<user id="001" role="admin">Taro</user>
```

- 開始タグの中に `属性名="値"` の形で書く
- 要素に**メタ情報**を付加する役割
- 値は必ず `"..."` または `'...'` で囲む

#### 要素 vs 属性（どちらで表現するか）

```xml
<!-- 属性で表現 -->
<user id="001">Taro</user>

<!-- 要素で表現 -->
<user>
  <id>001</id>
  <name>Taro</name>
</user>
```

> 一般的な使い分け：**構造的なデータ → 要素 / 識別子・補足情報 → 属性**

---

### 1-3-4. XML の厳密なルール（Well-formed XML）

```xml
<!-- NG：タグが閉じていない -->
<name>Taro

<!-- NG：入れ子が崩れている -->
<user><name>Taro</user></name>

<!-- NG：ルート要素が複数 -->
<user>Taro</user>
<user>Hanako</user>

<!-- OK：ルート要素で包む -->
<users>
  <user>Taro</user>
  <user>Hanako</user>
</users>
```

---

### 1-3-5. 特殊文字のエスケープ

| 文字 | エスケープ | 理由 |
|------|-----------|------|
| `<` | `&lt;` | タグの開始と誤認される |
| `>` | `&gt;` | タグの終了と誤認される |
| `&` | `&amp;` | エンティティ参照の開始と誤認される |
| `"` | `&quot;` | 属性値の区切りと誤認される |
| `'` | `&apos;` | 属性値の区切りと誤認される |

> ⚠️ **重要**：`&` で始まり `;` で終わる記法 → これが次章「**エンティティ**」の入口

## 1-4. XML 処理の全体像

```
XMLデータ → [XMLパーサー] → アプリケーションが使えるデータ構造
```

> **XXE 攻撃は「パーサーが処理する段階」で起きる**。パーサーが何を・どう処理するかが攻撃の核心。

## 1-5. 自身の認識

### 認識（原文）

> XMLは、データを意味のある構造で表現するための言語である。タグでデータの構成や構造を定義することにより、コンピューターがデータ構造の定義を理解できるようになる。JSONのようなデータの伝達の役割を果たすが、JSONとの差別化点は、タグによる属性値の付加が可能であるため、要素となるデータの詳細な情報を記述できる点である。

### 評価

| 認識の内容 | 正誤 | コメント |
|-----------|------|---------|
| データを意味のある構造で表現する言語 | ✅ 正しい | 本質を捉えている |
| タグでデータ構造を定義し、コンピューターが理解できる | ✅ 正しい | 正確 |
| JSON のようなデータ伝達の役割 | ✅ 正しい | 正確 |
| JSON との差別化点 = 属性値の付加 | ⚠️ 不正確 | 補足が必要（下記参照） |

### 補足1：属性値は差別化の本質ではない

JSON でも同様の情報を表現できる：

```json
{ "user": { "id": "001", "name": "Taro" } }
```

```xml
<user id="001">Taro</user>
```

→ 表現できる情報量に本質的な差はない。

### 補足2：「コンピューターが理解する」の主体はパーサー

「コンピューター」ではなく、より正確には **XML パーサー** というソフトウェアがタグを解析・解釈する。

### ✅ 修正版の認識

> XML は、タグでデータ構造を定義し、**XML パーサーを通じて**アプリケーションが解釈できる形にするデータ表現言語である。JSON との差別化点は属性値だけでなく、**DTD によるスキーマ定義・バリデーション機能**にあり、これが XXE の核心につながる。

## 1-6. 次章への橋渡し

- エスケープ記法（`&lt;` など）で登場した `&〜;` の記法
- これは **エンティティ参照** と呼ばれる
- エンティティを定義する仕組みが **DTD**
- DTD が XXE 攻撃の直接の舞台となる

→ **次章：DTD（Document Type Definition）**
