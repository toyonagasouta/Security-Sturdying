# 第6章：アプリケーションの信頼境界

## 6-1. 信頼境界とは何か

**信頼境界（Trust Boundary）** とは、「信頼できるデータ」と「信頼できないデータ」の境界線。

```
【信頼できる領域】              【信頼できない領域】
アプリ内部のデータ    ←境界線→    外部から来るデータ
DB の値                           ユーザーの入力
設定ファイル                       HTTP リクエスト
内部 API の結果                    XML データ  ← XXE はここ
```

> **核心**：外部から来たデータを「そのまま信頼して処理する」ことが、
> インジェクション系脆弱性すべての根本原因。

---

## 6-2. XXE における信頼境界の問題

XXE を信頼境界の観点で整理すると、問題の構造が明確になる。

```
攻撃者
    ↓ 細工した XML を送信（信頼できない領域）
    ↓
──────────────── 信頼境界 ────────────────
    ↓
アプリが XML をパーサーに渡す
    ↓
パーサーが DOCTYPE・エンティティを「信頼して」処理する  ← ここが問題
    ↓
サーバーが file:// や http:// へアクセスする
    ↓
内部リソース（信頼できる領域）への侵入が成立する
```

> パーサーは「受け取った XML の DOCTYPE も含めて正当な命令である」と信頼して処理する。
> 外部から来たデータが、内部の処理系（パーサー）に直接命令を与えられる状態が脆弱性の本質。

---

## 6-3. インジェクション系脆弱性に共通する構造

XXE は「インジェクション系脆弱性」の一種。
信頼境界の問題は XXE だけでなく、他の脆弱性にも共通する普遍的な構造。

| 脆弱性 | 外部から来るデータ | 侵入する処理系 |
|--------|-----------------|--------------|
| **XXE** | XML（DTD・エンティティ） | XML パーサー |
| **SQL インジェクション** | ユーザー入力 | SQL エンジン |
| **XSS** | ユーザー入力 | ブラウザの HTML/JS エンジン |
| **コマンドインジェクション** | ユーザー入力 | OS のシェル |

```
共通する構造：
外部からの「データ」が
処理系への「命令」として解釈されてしまう
```

> XXE の理解は、インジェクション系脆弱性全体の理解につながる。

---

## 6-4. 信頼境界を破る3つの条件

診断の観点で「なぜ脆弱なのか」を判断するための条件。

```
条件1：外部からのデータを受け取る口がある
            ↓
        XML を受け付けるエンドポイントが存在する

条件2：受け取ったデータを検証・サニタイズせずに処理する
            ↓
        DTD・エンティティの内容をチェックしない

条件3：処理系が外部データの命令を「信頼して実行」する
            ↓
        パーサーが外部エンティティをデフォルトで処理する
```

> この3条件がすべて揃ったとき、信頼境界は破られ XXE が成立する。

---

## 6-5. 入力値の検証・サニタイズという概念

信頼境界を守るための基本的な考え方。

### 検証（Validation）

「受け取ったデータが期待する形式・内容かどうか」を確認すること。

```
XXE における検証の例：
・DOCTYPE 宣言が含まれていないか確認する
・SYSTEM キーワードが含まれていないか確認する
・XML の構造が期待通りかスキーマで検証する
```

### サニタイズ（Sanitize）

「受け取ったデータから危険な部分を除去・無害化」すること。

```
XXE におけるサニタイズの例：
・DOCTYPE 宣言を取り除く
・エンティティ参照を無効化する
・危険な URI スキーム（file:// など）を除去する
```

### 検証・サニタイズの限界と根本対策

```
検証・サニタイズ
    ↓
パターンの網羅漏れが生じる可能性がある
（想定外の記法・エンコードで回避されることがある）
    ↓
より根本的な対策
    ↓
「パーサー自体が外部エンティティを処理しない設定にする」
    ↓
信頼境界の入口ではなく、処理系そのものを制限する
```

> 検証・サニタイズは補助的な対策として有効だが、
> パーサーの機能を無効化するほうが確実な対策。

---

## 6-6. 「デフォルト信頼」という設計思想の問題

XXE が広く存在する背景には、パーサーの設計思想の問題がある。

```
【安全でない設計（多くの既存パーサー）】
外部エンティティ処理：デフォルト有効
    → 開発者が意識しなければ脆弱なまま動く
    → 「何もしない」 = 脆弱

【安全な設計（Secure by Default）】
外部エンティティ処理：デフォルト無効
    → 必要なときだけ明示的に有効化する
    → 「何もしない」 = 安全
```

> **Secure by Default**（デフォルトで安全）という原則。
> 診断時に「デフォルト設定のまま使っていないか」を確認することが重要。

## 6-7. 信頼境界の観点で見た XXE の全体像

```
【攻撃者】
    ↓ 細工した XML を送信
──────── 信頼境界（本来ここで止めるべき）────────
    ↓ 境界を突破（DTD・エンティティが検証されない）
【XML パーサー】
    ↓ 外部エンティティを「信頼して」展開
【サーバー内部】
  ├─ /etc/passwd を読む
  ├─ 内部ネットワークへアクセスする
  └─ クラウドメタデータを取得する
    ↓
【攻撃者】が内部情報を取得
```

## 6-8. 診断時の確認ポイント

信頼境界の概念を診断に活かすための確認事項。

| 確認項目 | 脆弱な状態 | 対策済みの状態 |
|---------|-----------|-------------|
| XML の受け口 | XML を受け付けるエンドポイントが存在する | — |
| DOCTYPE の処理 | DOCTYPE 宣言をそのまま処理する | DOCTYPE を拒否する設定がある |
| 外部エンティティ | デフォルト有効のパーサーをそのまま使用 | 外部エンティティ処理を無効化済み |
| 入力検証 | 受け取った XML を無検証でパーサーに渡す | スキーマ検証・DOCTYPE の除去を実施 |
| エラー情報 | エラーにファイル内容や内部パスが含まれる | エラー内容を外部に返さない |

## 6-9. 第6章まとめ

```
信頼境界
├─ 定義：「信頼できるデータ」と「信頼できないデータ」の境界線
│
├─ XXE における問題
│   └─ 外部から来た XML の DOCTYPE・エンティティを
│      パーサーが「信頼して実行」してしまう
│
├─ インジェクション系脆弱性に共通する構造
│   └─ 外部の「データ」が処理系への「命令」として解釈される
│      （XXE・SQLi・XSS・コマンドインジェクション すべて同じ構造）
│
├─ 信頼境界を守る手段
│   ├─ 検証（Validation）  ：期待通りのデータかチェック
│   ├─ サニタイズ（Sanitize）：危険な部分を除去
│   └─ Secure by Default  ：処理系自体をデフォルト無効に設定（最も確実）
│
└─ 診断の視点
    └─ 「外部データが検証なく処理系に渡っていないか」を確認する
```
