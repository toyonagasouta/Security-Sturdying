# HTTPプロトコル基礎 復習まとめ

## 1. リクエスト/レスポンスの構造

### あなたの認識
- リクエスト・レスポンス共通で、ヘッダーとボディの境界は「空行」で区切られる
- リクエスト = リクエストライン + ヘッダー + 空行 + ボディ
- レスポンス = ステータスライン + ヘッダー + 空行 + ボディ

### 判定:正しい

### 実際の仕組み
HTTP/1.1ではテキストベースで以下の形式に統一されている。

**リクエスト例**
```
GET /search?q=test HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Cookie: session=abc123

(ボディ。GET等では通常なし)
```

**レスポンス例**
```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 145
Set-Cookie: session=xyz789

{"result": "success"}
```

- 空行(CRLF CRLF)はヘッダー終端を示す唯一の方法であり、ボディの開始位置を機械的に判定するための区切り
- ボディが省略される場合(GET、204 No Contentなど)もあるが、構造のルール自体は変わらない

**補足**: HTTP/2はバイナリフレーム形式のため、上記のテキスト構造そのままではない(下記7章参照)。Burpはデコードして見やすい形式で表示している。

## 2. パスとクエリ文字列

### あなたの認識
- `?`より前がパス、`?`以降がクエリ文字列(パラメータ)
- パスはサーバが用意しているAPIの識別値であり、これを参照してAPIが呼び出され処理が行われる
- クエリ文字列は特定したAPIのロジック内で処理されるパラメータ

### 判定:概ね正しい(「パス=API識別値」は一部正確化が必要)

### 実際の仕組み

**例**
```
GET /.well-known/device-bound-sessions?experiment_id=l2_enabled HTTP/1.1
```
- パス: `/.well-known/device-bound-sessions`
- クエリ文字列: `experiment_id=l2_enabled`(複数ある場合は`&`で連結)

**パスについての正確な理解**:
パスは「APIの識別値」に限らず、より広く「サーバ内の何らかの処理または資源を指し示す識別子」。大きく2パターンに分かれる。

1. **静的ファイルへのマッピング**: パスがサーバ上の実ファイル(画像・HTML等)に対応し、そのまま返される
2. **動的処理へのルーティング**: パス(+メソッド)に応じて、対応するプログラム(コントローラー)が実行される

**クエリ文字列**: サーバ側のプログラムコード内で取り出され(例: Flaskの`request.args.get('experiment_id')`)、条件分岐やDB検索条件などに使われる。あなたの認識通りで正確。

**全体の流れ**:
```
1. ブラウザ: GET /search?q=tokyo HTTP/1.1 を送信
2. Webサーバ: リクエストを受信
3. ルーティング: パス"/search"に対応する処理を探す
4. 該当処理が実行され、クエリパラメータ"q=tokyo"を取得
5. その値を使ってDB検索などを実行
6. 結果をレスポンスとして返す
```

**注意点**: クエリ文字列はURLの一部として見える形で送信されるため、機密情報を入れるのは避けるべき。POSTの場合パラメータは通常ボディに入り、リクエストラインには表示されない。

## 3. ルーティング/コントローラーの仕組み

### あなたの認識
- ルーター・ルーティングはネットワーク機器ではなく、バックエンドの構成要素(コントローラー)のこと
- API名とその処理先を対応付けたファイルを用意しておき、リクエスト内容から該当するAPIを探し出し、対応する処理を実行する、というのが現代の動的処理の仕組み

### 判定:正しい

### 実際の仕組み
フレームワーク(Flask, Django, Express, Spring等)が「パス ⇔ 対応する処理(関数)」の対応表(ルーティングテーブル)を持つ。

```python
# Flaskの例
@app.route('/users')
def get_users():
    # ユーザー一覧を返す処理
    ...

@app.route('/users/<id>')
def get_user(id):
    # 特定ユーザーを返す処理
    ...
```

**重要なポイント**: 処理は「パス」だけでなく「パス + メソッド」の組み合わせで決まる。
- `/users`へのGET → 一覧取得
- `/users`へのPOST → 新規作成

のように、同じパスでもメソッドによって呼ばれる処理(コントローラー)が異なるのが一般的な設計。

## 4. 静的ファイル配信とパストラバーサル

### あなたの認識
- 静的ファイルへのマッピングは、リクエストに対してサーバのリソース上にあるファイルを渡す際に利用される
- 渡すリソースの内容に制限を設けないことで攻撃される

### 判定:正しい(脆弱性名: パストラバーサル)

### 実際の仕組み
ユーザーから渡されたパスをそのままファイルシステムのパスに変換してファイルを読み込む実装の場合に発生する。

```python
# 脆弱な実装例
@app.route('/files/<filename>')
def get_file(filename):
    return open(f'/var/www/uploads/{filename}').read()
```

**通常のリクエスト**:
```
GET /files/photo.jpg HTTP/1.1
→ /var/www/uploads/photo.jpg を返す(意図通り)
```

**攻撃リクエスト**:
```
GET /files/../../../../etc/passwd HTTP/1.1
→ /var/www/uploads/../../../../etc/passwd
→ 結果的に /etc/passwd を返してしまう
```

`../`(1階層上に戻る)を繰り返すことで、本来アクセスできないファイル(設定ファイル、パスワードファイル、ソースコード等)を読み取られる可能性がある。

**対策**:
- 渡されたファイル名から`../`等の危険な文字列を除去・検証
- アクセス許可ディレクトリを限定し、そこから出られないようにする
- ファイル名を直接使わず、IDなどに変換してから内部でパスを解決

**Burpでの診断手法**: `../`を含むパラメータを試して、レスポンス内容(エラーかファイル内容が返るか)を確認する。

## 5. メソッド

### あなたの認識
- メソッドはリクエストでサーバ側に要求する内容であり、リクエストラインに記述される
- GET: ファイルやパラメータ等のリソースを取得
- POST: サーバ側に何らかの処理を要求する
- DELETE: リソースの削除
- PUT: リソースの更新
- PATCH: 部分的な更新

### 判定:概ね正しい(POSTの説明に補足必要)

### 実際の仕組み・補足

| メソッド | 説明 |
|---|---|
| GET | リソースの取得。認識通り |
| POST | 「サーバ側に何らかの処理を要求する」という認識でも大枠は正しいが、より正確には「新しいリソースの作成」や「データの送信(フォーム送信等)」が主目的。冪等性(同じリクエストを何度送っても結果が同じか)がない点がGETとの大きな違い |
| PUT | リソースの更新、または存在しない場合は作成。送ったデータでリソース全体を置き換えるイメージ。冪等性あり |
| PATCH | 部分的な更新。認識通り |
| DELETE | リソースの削除。認識通り |

### 今後学習すべき項目(言及していなかったメソッド)
- **HEAD**: GETと同じだがボディを返さない(ヘッダーだけ確認したい時に使う)
- **OPTIONS**: そのリソースがサポートするメソッドを確認する。CORSのプリフライトリクエストで使われる(7章のCORS参照)

## 6. ステータスコード

### あなたの認識
- ステータスコードはレスポンスのステータスラインに表示されるコード
- 100番台: 処理の実行中
- 200番台: 処理の成功
- 300番台: サービスサーバのプログラム処理へのリダイレクト
- 400番台: クライアント側のエラー
- 500番台: サーバ側のエラー

### 判定:300番台のみ修正が必要。修正後は正しい

### 実際の仕組み

| 番台 | 意味 |
|---|---|
| 1xx | 情報提供。処理が継続中であることを示す(例: 100 Continue) |
| 2xx | 成功 |
| 3xx | **クライアントに対して「別のURLにアクセスしてください」と指示するもの**。サーバ内部処理の話ではない |
| 4xx | クライアント側のエラー(例: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found) |
| 5xx | サーバ側のエラー(例: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable) |

### 3xx(リダイレクト)の仕組み詳細
300番台のレスポンスを受け取ると、ブラウザはレスポンスヘッダーの`Location`に記載されたURLへ**自動的に次のリクエストを送信**する。

```
HTTP/1.1 302 Found
Location: https://example.com/login
```

このレスポンスを受け取ったブラウザは、自動的に`https://example.com/login`へGETリクエストを送る。

**主なコード**:
- **301 Moved Permanently**: リソースが永久に移動(検索エンジンはリンクを更新)
- **302 Found / 303 See Other**: 一時的なリダイレクト
- **304 Not Modified**: リソースに変更なし。キャッシュを使用(`Location`なし、ボディも返らない)

**Burpでの注意点**: 自動リダイレクト追従(Follow redirections)の設定により、302レスポンス自体(Locationヘッダー、Set-Cookieの内容)を見落とすことがある。診断時は自動追従を切って確認することが重要(認証バイパス等の発見につながる場合がある)。

## 7. ヘッダー・HTTP/1.1とHTTP/2の違い(今後学習すべき内容)

ここから先は、自身で詳しい言及をしていなかったため、説明内容をまとめたもの。今後の学習項目として整理。

### 7-1. Cookieの仕組み

#### あなたの認識
- CookieとはユーザーのセッションID情報をサーバに伝える事でセッションの状態を管理する役割をもつヘッダー値
- ユーザーが認証に成功した際、サーバ側が保存期間を設けたセッションIDをランダムに作成し、ユーザーのアカウント情報に紐づける
- 作成したセッションIDをレスポンスで返す事で、ブラウザ側でセッションIDをCookieとして保存する
- リクエストの際にセッションIDをヘッダー情報に付加してサーバに渡すことで、サーバはセッションIDに紐づけたアカウントに対する操作を行う

#### 判定:概ね正しい(2点補足が必要)

#### 実際の仕組み

**補足1: CookieはセッションID専用の仕組みではない**
Cookieは「ブラウザにキーと値のペアを保存させ、以降のリクエストで自動的にサーバへ送り返す仕組み」であり、セッション管理は代表的な用途の一つ。他にも言語設定、広告トラッキングID、A/Bテストのグループ情報、CSRF対策トークンなどに使われる。

**補足2: セッションIDとアカウント情報の関係**
セッションIDそのものにアカウント情報は含まれていない。サーバ側のセッションストア(DBやメモリキャッシュ等)の検索キーとして機能する。

```
サーバ側: セッションストア
┌──────────────────────────────────────┐
│ セッションID(key) → アカウント情報(value) │
│ abc123 → {user_id: 42, role: admin}    │
└──────────────────────────────────────┘
```

**全体の流れ(正確版)**
```
1. ユーザーがID/パスワードでログイン(POST)
2. サーバが認証成功を確認
3. サーバがランダムなセッションIDを生成し、セッションストアに
   「セッションID → アカウント情報」として保存
4. レスポンスのSet-CookieヘッダーでセッションIDをブラウザに渡す
   Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax
5. ブラウザがCookieとしてセッションIDを保存
6. 以降のリクエストでブラウザが自動的にCookieヘッダーに付加して送信
   Cookie: session_id=abc123
7. サーバがセッションIDをキーにセッションストアを検索し、
   対応するアカウント情報を取り出して処理
```

**知っておくべきCookie属性**:
- `HttpOnly`: JavaScriptからアクセス不可(XSS対策)
- `Secure`: HTTPS通信でのみ送信
- `SameSite`: クロスサイトでCookieを送るかどうか(CSRF対策に関連。Strict/Lax/None)

**Burpでの観点**:
- Cookieのセッションを他ユーザーのIDに差し替えた場合、なりすましが成立するか
- ログアウト後にセッションIDが無効化されているか
- HttpOnly/Secure/SameSiteの各属性がついているか

### 7-2. Authorizationヘッダーの仕組み

#### Cookieとの違い
- **Cookie**: ブラウザが該当ドメインへのリクエスト時に自動的に`Cookie`ヘッダーへ付加する(ブラウザ組み込みの仕組み)
- **Authorization**: ブラウザが自動で付加する仕組みを持たない。サーバから取得したJavaScriptファイルのコードが、明示的にトークンを取り出してヘッダーに付加する必要がある(アプリケーションコード側での実装が必要)

```javascript
// Authorizationは開発者がコードで明示的に付加する
fetch('/api/users', {
  headers: {
    'Authorization': `Bearer ${localStorage.getItem('token')}`
  }
});
```

※モバイル/デスクトップアプリの場合は、サーバから取得するのではなくアプリ自体にそのコードが含まれている。

#### CookieとAuthorizationの使い分け
サーバの要件によって使い分けられる。両方を組み合わせて使うサービスも多い。

| | Cookie+セッション | Authorization(JWT等) |
|---|---|---|
| 主な用途 | Webブラウザ向けサービス | API(モバイルアプリ、SPAなど) |
| トークン保存場所 | ブラウザが自動管理 | アプリが自前で管理(localStorage等) |
| サーバの状態 | セッションDBが必要(ステートフル) | 不要(ステートレス) |
| 複数サーバへの対応 | セッションDB共有が必要 | トークン検証だけなので容易 |

#### 認証スキームの種類
Authorizationヘッダーの値の種類は「認証スキーム」と呼ばれる。

**1. Basic認証**
```
Authorization: Basic dXNlcjpwYXNzd29yZA==
```
`user:password`をBase64エンコードしたもの。Base64は暗号化ではなく単なる変換であり、Burpで簡単にデコードできる。現代ではほぼ使われない。

**2. Bearer認証(トークン認証)**
```
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```
現代のAPIで最も使われる形式。Bearerは「トークンを運ぶ形式」であり、トークンの形式がJWTとは限らない(独自のランダムトークンの場合もある)。ただし実装上はBearerトークン=JWTのケースが非常に多い。

#### JWT(JSON Web Token)の仕組み

JWTは`.`で区切られた3つの部分から構成される。

```
ヘッダー.ペイロード.署名
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0Mn0.署名部分
```

```
ヘッダー  → {"alg": "HS256", "typ": "JWT"}          ← 署名アルゴリズムの種類
ペイロード → {"sub": "42", "role": "admin", "exp": ...} ← ユーザー情報(平文)
署名      → ヘッダー+ペイロードを鍵で署名したもの
```

**署名アルゴリズムの2方式**:

| 方式 | 鍵の使い方 | 特徴 |
|---|---|---|
| HMAC(HS256等) | 署名・検証ともに同じ秘密鍵(共通鍵) | 鍵が1つ。漏洩すると偽造され放題。小規模システム向け |
| RSA/ECDSA(RS256/ES256等) | 秘密鍵で署名、公開鍵で検証 | 公開鍵は誰に渡しても良い。大規模システム向け |

```
署名生成: ヘッダー + ペイロード → 秘密鍵で署名 → 署名値
署名検証: ヘッダー + ペイロード + 署名値 → 公開鍵で検証 → 正当かどうか
```

**重要: Base64エンコードは暗号化ではない**
```
暗号化: 鍵がなければ元のデータを復元できない → 機密性がある
Base64: 決まったルールで変換するだけ → 誰でも簡単に元に戻せる
```
ヘッダーとペイロードはBase64エンコードされているだけなので、Burpの「Decoder」タブでデコードすれば中身が丸見えになる。したがってJWTのペイロードには、パスワードはもちろんメールアドレス等の機密情報も入れてはいけない。署名は「改ざんされていないことの証明」であり「内容を隠す仕組み」ではない点が重要。

**全体フロー**
```
1. ユーザーがID/パスワードでログイン
2. サーバが認証成功を確認
3. サーバがJWTを生成
   - ヘッダー: {"alg": "RS256", "typ": "JWT"}
   - ペイロード: {"sub": "42", "role": "admin", "exp": ...}
   - 署名: ヘッダー+ペイロードを秘密鍵で署名
   - 3つをBase64エンコードして"."で連結
4. JWTをレスポンスでクライアントへ返す
5. クライアントがJWTを保存(localStorageなど)
6. 以降のリクエストでAuthorizationヘッダーに付加
   Authorization: Bearer eyJhbGci...
7. サーバがJWTを受け取り:
   a. 署名を検証(改ざんされていないか)
   b. exp(有効期限)を確認
   c. ペイロードからユーザー情報を取り出して処理
```

**Burpでの観点**:
- ペイロードの`role`を改ざんしてサーバが署名検証をきちんと行っているか確認
- `exp`を未来に改ざんして有効期限チェックを回避できるか
- `alg`を`none`に変更して署名検証をスキップできるか(alg:none攻撃)
- RS256使用時に`alg`をHS256に変更し、公開鍵を秘密鍵として悪用できるか(アルゴリズム混同攻撃)
- BurpのJWT Editor拡張で解析・改ざんが容易にできる

### 7-3. Content-Typeの仕組み

#### 役割
リクエスト・レスポンス両方で使われ、ボディのデータ形式を受け取る側に伝える。ボディがある場合にのみ意味を持つ(GETや204 No Contentなど、ボディがない場合は通常不要)。

```
リクエストのContent-Type: 「私が送るボディはこの形式です」とサーバに伝える
レスポンスのContent-Type: 「私が返すボディはこの形式です」とブラウザに伝える
```

#### データの解釈の仕組み
コンピュータ上のデータは突き詰めると0と1のビット列であり、同じビット列でも「どう解釈するか」で全く異なるものになる。Content-Typeは「このバイト列はこのルールで解釈してください」という指示書の役割を果たす。

ブラウザは内部に複数の処理エンジンを持ち、Content-Typeに応じて担当エンジンに処理を渡す。
```
受信したレスポンス
└── Content-Typeを確認
    ├── text/html        → HTMLパーサー → DOMツリー構築 → 描画
    ├── application/json → JSONパーサー → JSオブジェクトに変換
    ├── image/jpeg       → 画像デコーダー → ピクセルデータに変換 → 描画
    ├── application/pdf  → PDFビューアに渡す
    └── application/octet-stream → OSのダウンロードマネージャーに渡す
```
「Content-Typeは処理の担当エンジンを振り分けるルーター的な役割」という理解が実用上最も正確。

#### 攻撃との関係
「Content-Typeの偽装」と「サーバ側の検証不足」の組み合わせが脆弱性の本質。

**ファイルアップロード脆弱性の例**:
```
攻撃リクエスト:
Content-Type: image/jpeg  ← 画像のふりをしている
ボディ: <?php system($_GET['cmd']); ?>  ← 実際はPHPコード
```
サーバがContent-Typeだけを信頼してファイルの中身を検証しない場合、PHPファイルがアップロードされ実行されてしまう。

**Content-Typeスニッフィングの例**:
```
サーバ: Content-Type: text/plain で返す
ボディ: <script>alert(1)</script>
ブラウザが中身を推測してHTMLとして解釈 → JavaScriptが実行される → XSS
```
これを防ぐのが`X-Content-Type-Options: nosniff`(ブラウザに推測せずContent-Typeに従うよう強制する)。

#### 主なContent-Typeの値

| 値 | 意味 | よく使われる場面 |
|---|---|---|
| `application/json` | JSON形式 | REST API |
| `application/x-www-form-urlencoded` | フォームデータ | HTMLフォームのPOST |
| `multipart/form-data` | ファイルを含むフォーム | ファイルアップロード |
| `text/html` | HTML | ページのレスポンス |
| `text/plain` | プレーンテキスト | シンプルなテキスト |
| `image/jpeg` `image/png` | 画像 | 画像ファイル |

**Burpでの観点**: リクエストのContent-Typeを変更してサーバの挙動変化を確認する(例: `application/x-www-form-urlencoded`のボディを`application/json`形式に書き換えて送る)。

### 7-4. CORSの仕組み

#### オリジンとは
スキーム・ホスト名・ポート番号の3つの組み合わせ。1つでも異なればクロスオリジン。
```
https://example.com:443
└─┬─┘ └────┬────┘ └┬┘
  │        │       └── ポート番号
  │        └────────── ホスト名
  └─────────────────── スキーム
```

#### なぜCORSが必要か(同一オリジンポリシー)
ブラウザには「同一オリジンポリシー(SOP)」が組み込まれており、あるオリジンのJavaScriptが別オリジンのリソースに自由にアクセスすることを禁止している(悪意あるサイトがユーザーのブラウザを踏み台にして、ログイン中の別サービスのAPIを叩くことを防ぐため)。CORSは「SOPの制限を、サーバが意図的に緩める仕組み」。

#### あなたの認識
「ブラウザがオリジンAのJSの代わりにオリジンBへ許可を求め、許可されれば本命のリクエストを送り、取得した内容をオリジンAのJSに渡す」という流れの認識は正確。

#### 仕組みの詳細

**シンプルなケース(プリフライトなし)**
プリフライトが省略される条件: メソッドがGET/POST/HEADのいずれか、かつヘッダーがブラウザ標準のもののみ。
```
1. ブラウザがOriginヘッダーを自動付加してリクエスト
   GET /data HTTP/1.1
   Origin: https://app.example.com
2. サーバがアクセス許可を返す
   Access-Control-Allow-Origin: https://app.example.com
3. ブラウザが許可を確認し、JavaScriptがレスポンスを読める
```

**プリフライトありのケース(OPTIONSメソッド)**
PUTやカスタムヘッダー(Authorization等)を含むリクエストの場合に発生。現代のAPIはAuthorizationヘッダーやapplication/jsonを使うことが多いため、プリフライトが発生するケースが多い。
```
1. ブラウザが事前確認を送る
   OPTIONS /api/data HTTP/1.1
   Origin: https://app.example.com
   Access-Control-Request-Method: POST
   Access-Control-Request-Headers: Authorization, Content-Type
2. サーバが許可内容を返す
   Access-Control-Allow-Origin: https://app.example.com
   Access-Control-Allow-Methods: GET, POST, PUT
   Access-Control-Allow-Headers: Authorization, Content-Type
   Access-Control-Max-Age: 86400
3. ブラウザが本来のリクエストを送る
   POST /api/data HTTP/1.1
   Authorization: Bearer eyJ...
```

#### 主なCORS関連ヘッダー

**リクエスト側(ブラウザが自動付加)**

| ヘッダー | 意味 |
|---|---|
| `Origin` | どのオリジンからのリクエストか |
| `Access-Control-Request-Method` | プリフライト時。本番で使うメソッド |
| `Access-Control-Request-Headers` | プリフライト時。本番で使うヘッダー |

**レスポンス側(サーバが返す)**

| ヘッダー | 意味 |
|---|---|
| `Access-Control-Allow-Origin` | アクセスを許可するオリジン |
| `Access-Control-Allow-Methods` | 許可するメソッド |
| `Access-Control-Allow-Headers` | 許可するヘッダー |
| `Access-Control-Allow-Credentials` | Cookie等の認証情報を含むリクエストを許可するか |
| `Access-Control-Max-Age` | プリフライト結果のキャッシュ時間 |

#### 重要な誤解ポイント
CORSはリクエスト自体をブロックする仕組みではない。
```
実際の動作:
リクエストはサーバに届く → サーバは処理する → レスポンスを返す
→ ブラウザがCORSヘッダーを確認
→ 許可されていなければJavaScriptがレスポンスを読めないようにする
```
CORSはあくまでも**ブラウザの機能**であり、Burp Suiteのようなブラウザを介さないツールからのリクエストはCORSの制限を受けない。

#### CORSの誤設定(セキュリティ上の問題)

**問題設定1: ワイルドカード**
```
Access-Control-Allow-Origin: *
→ 全てのオリジンからのアクセスを許可してしまう
```

**問題設定2: Originヘッダーをそのまま返す**
```
リクエスト: Origin: https://evil.com
レスポンス: Access-Control-Allow-Origin: https://evil.com
→ 攻撃者のオリジンを動的に許可してしまう
```

**問題設定3: ワイルドカード + Credentials**
```
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
→ 仕様上無効だが、実装ミスで機能してしまう場合がある
```

**Burpでの観点**: `Origin`ヘッダーを任意の値に書き換え、レスポンスの`Access-Control-Allow-Origin`がどう変わるかを確認する。

### 7-5. ヘッダー学習の到達目安

1. **Cookie/Authorization**: 認証・セッション管理がどう行われているかを読み取れる。Burpで値を改ざんして挙動を確認する発想ができること
2. **Content-Type**: リクエスト/レスポンスのボディがどう解釈されているかを把握できること
3. **CORS**: なぜこのヘッダーが付いているのか、このサイトは他オリジンからのアクセスをどこまで許可しているかを読み取れること

### 7-6. HTTP/1.1とHTTP/2の違い

「データの表示形式(バイナリ vs テキスト)の違い」は本質ではなく、それは結果(副産物)。本質的な違いは以下。

---

#### 1. 多重化(Multiplexing) ※最重要

#### あなたの認識
- HTTP/1.1では通信の受信応答が逐次処理であった
- HTTP/2では、TCP接続の中でストリームという論理的な通り道を複数作ることにより、データの送受信を並列に行うことが可能になった
- リクエストとレスポンスのペアをストリーム単位で設けて、リクエスト/レスポンスデータをフレームに分割した際、そのストリームIDに紐づけることで、通信データの識別を可能にする。結果的に並列での通信を行える

#### 判定:正しい

#### 実際の仕組み

**HTTP/1.1の問題: Head-of-Line Blocking**

HTTP/1.1では1本のTCP接続内でリクエストとレスポンスは「順番待ち」になる。

```
1本のTCP接続:
リクエスト1送信 → レスポンス1を待つ → レスポンス1受信
                                    → リクエスト2送信 → ...
```

**ブラウザの回避策(HTTP/1.1時代)**:
複数本のTCP接続を同時に張ることで擬似的に並列化していた(通常、同一ホストに対して6本前後)。ただしTCP接続/TLSハンドシェイクのコストが高く、接続数にも上限があるためボトルネックになりやすかった。

**HTTP/2の解決策: ストリームとフレーム**

```
ストリーム  = リクエスト/レスポンスのペアを通す論理的な通り道(並列処理の単位)
フレーム    = 実際に送信される最小単位のデータ(HEADERS、DATA等)
ストリームID = どのフレームがどのストリームに属するかを識別するための紐付け情報
```

```
1本のTCP接続
├── ストリーム1: リクエストA / レスポンスA
├── ストリーム3: リクエストB / レスポンスB
└── ストリーム5: リクエストC / レスポンスC
(これらが同時並行で進む)

実際の送信イメージ(フレームが入り混じって送られる):
[ストリーム1のHEADERSフレーム]
[ストリーム3のHEADERSフレーム]
[ストリーム1のDATAフレーム]
[ストリーム5のHEADERSフレーム]
...
```

**注意点**: HTTP/2はHTTP層では多重化を実現したが、その下のTCP層では依然としてHead-of-Line Blockingが残る(パケットロスが起きると関係ないストリームも待たされる)。これを完全解決するのがHTTP/3(QUICプロトコル、UDPベース)。

**Burpでの観察ポイント**: HTTP/2接続サイトではほぼ同じタイムスタンプで複数リクエストがHTTP historyに記録される。これは複数ストリームが同時進行している証拠。

---

#### 2. ヘッダー圧縮(HPACK)

#### あなたの認識
- サーバ・クライアント間の通信において、毎回のリクエスト/レスポンスで共通するヘッダー情報を圧縮することで、送信するデータ量を減らし、通信のオーバーヘッドを減らすことを目的とした仕組み
- 静的テーブル: よく使われるヘッダー名や値のペアをあらかじめ番号付きの表としてHTTPの規格に定義しておき、リクエスト/レスポンス通信で参照する方法
- 動的テーブル: 送信側と受信側のそれぞれが、通信で利用したヘッダー情報を自身の学習済みテーブルに保存しておき、テーブル番号として紐づけておくことで、以降の通信で番号として参照する方法
- 圧縮アルゴリズムが使われる場合、入力に対する圧縮後の出力を試行を重ねて予測するCRIME攻撃が理論上可能

#### 判定:正しい

#### 実際の仕組み

**なぜ必要か**: HTTP/1.1では同じCookieやUser-Agentなどが毎回フルテキストで送られる。1ページの読み込みで何十回もリクエストが発生する現代のWebサイトでは、この繰り返しが無視できないオーバーヘッドになる。

**圧縮の仕組み**:

```
静的テーブル(規格に定義済み):
番号2 → :method: GET
番号4 → :path: /
番号8 → :status: 200
(61個の定義済みエントリ)

動的テーブル(通信しながら学習):
1回目: Cookie: session=abc123 → フルテキストで送信 → 番号62として記録
2回目以降: 番号62 ← これだけ送れば良い
```

**イメージ図**:
```
HTTP/1.1(毎回フルテキスト):
リクエスト1: [Host: example.com][User-Agent: ...長い...][Cookie: ...長い...]
リクエスト2: [Host: example.com][User-Agent: ...長い...][Cookie: ...長い...]
              ↑ 毎回同じ内容を全部送る(無駄が多い)

HTTP/2 + HPACK:
リクエスト1: [Host: example.com][User-Agent: ...長い...][Cookie: ...長い...]
              → 動的テーブルに62, 63, 64番として記録
リクエスト2: [62][63][64] ← 番号の参照だけ(大幅に小さくなる)
```

**CRIME攻撃について**:
圧縮アルゴリズムは「同じ文字列が繰り返されるとサイズが小さくなる」特性を持つ。攻撃者が推測した値を含むリクエストを送り、圧縮後のサイズ変化を観察することで、本来見えないはずのCookieやトークンの値を1文字ずつ推測していく攻撃。HPACKでは機密性の高いヘッダーの扱いに緩和策が取られている。

---

#### 3. サーバープッシュ(廃止済み)

#### あなたの認識
- サーバがクライアントに必要だと判断したリソースを先に送りつける仕組み
- 理論上の通信回数が減るためオーバーヘッドの削減が可能
- ブラウザ側がキャッシュしているリソースをサーバが知る方法がないため重複送信が発生する
- ブラウザ側の実装が複雑になる割にパフォーマンス向上が限定的
- 多重化だけで十分に高速化できるという点から廃止された
- 現在は代替としてEarly Hints(103ステータスコード)というヒントだけを先に送る仕組みが注目されている

#### 判定:正しい

#### 補足

```
通常の流れ:
ブラウザ: GET /index.html
サーバ: index.htmlを返す
ブラウザ: index.htmlを解析 → style.cssが必要と判断
ブラウザ: GET /style.css ← ここで初めてリクエスト

サーバープッシュの想定:
ブラウザ: GET /index.html
サーバ: index.htmlを返す + style.cssも一緒に送りつける
ブラウザ: index.htmlを解析 → style.cssはすでに受け取っている
```

**Early Hints(103)との違い**:
サーバープッシュは「送りつける」のに対し、Early Hintsは「このリソースが必要になるよ」というヒントだけを先に返し、ブラウザに判断させる。ブラウザ側がキャッシュを持っていればヒントを無視できるため、無駄な重複送信の問題が発生しない。

---

#### 4. 暗号化との関係

#### あなたの認識
- HTTP/2では理論上は平文による通信が可能だが、実用上は平文で送ることはできない
- 現在実装されているブラウザはTLSを必須とした実装になっている
- HTTP/1.1は平文・暗号化どちらの通信も可能。HTTP/2は暗号化されたデータのみ通信を行う

#### 判定:正しい

#### 整理

```
HTTP/1.1 + 平文  = HTTP  (http://...)   ← 可能
HTTP/1.1 + TLS  = HTTPS (https://...)  ← 可能
HTTP/2   + TLS  = HTTPS (https://...)  ← 実用上これのみ
HTTP/2   + 平文 = h2c                  ← 仕様上可能だがブラウザ非対応
```

ブラウザがTLSを必須にした理由: HTTP/2導入のタイミングで「HTTPSをWebの標準にする」というブラウザベンダーの方針と合致したこと、またTLSのALPN拡張を使うとハンドシェイク中にHTTP/2対応を確認できてシンプルになることが挙げられる。

---

#### 5. リクエストラインの構造変化(補足)

- HTTP/2には「リクエストライン」という行自体が存在しない
- HTTP/1.1の`GET /path HTTP/1.1`は、HTTP/2では`:method`, `:path`, `:scheme`, `:authority`という擬似ヘッダー(pseudo-header)に分解される
- ヘッダー情報→HEADERSフレーム(HPACK圧縮されたバイナリ)、ボディ→DATAフレーム という形でバイナリのフレーム単位に分割して送信される
- Burpが`GET /path HTTP/2`のように表示するのは、本来バラバラな擬似ヘッダーやフレーム構造を、ツールがHTTP/1.1風の見慣れた形式に**再構成して表示**しているため(ワイヤー上の実データとは別物)

---

#### HTTP/1.1 vs HTTP/2 全体整理

| 項目 | HTTP/1.1 | HTTP/2 |
|---|---|---|
| データ形式 | テキスト | バイナリフレーム |
| 並列処理 | 複数TCP接続で擬似的に実現 | 1接続内でストリームにより実現 |
| ヘッダー | 毎回フルテキストで送信 | HPACKで圧縮(静的/動的テーブル) |
| サーバープッシュ | なし | あったが廃止(Early Hintsが代替) |
| 暗号化 | 任意(HTTP/HTTPS両対応) | 実用上HTTPS必須 |
