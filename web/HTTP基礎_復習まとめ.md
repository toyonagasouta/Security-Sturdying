## 1. リクエスト/レスポンスの構造

### あなたの認識
- リクエスト・レスポンス共通で、ヘッダーとボディの境界は「空行」で区切られる
- リクエスト = リクエストライン + ヘッダー + 空行 + ボディ
- レスポンス = ステータスライン + ヘッダー + 空行 + ボディ

### 判定:正しい

### 実際の仕組み
HTTP/1.1ではテキストベースで以下の形式に統一されている。

**リクエスト例**
```
GET /search?q=test HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0
Cookie: session=abc123

(ボディ。GET等では通常なし)
```

**レスポンス例**
```
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 145
Set-Cookie: session=xyz789

{"result": "success"}
```

- 空行(CRLF CRLF)はヘッダー終端を示す唯一の方法であり、ボディの開始位置を機械的に判定するための区切り
- ボディが省略される場合(GET、204 No Contentなど)もあるが、構造のルール自体は変わらない

**補足**: HTTP/2はバイナリフレーム形式のため、上記のテキスト構造そのままではない(下記7章参照)。Burpはデコードして見やすい形式で表示している。

## 2. パスとクエリ文字列

### あなたの認識
- `?`より前がパス、`?`以降がクエリ文字列(パラメータ)
- パスはサーバが用意しているAPIの識別値であり、これを参照してAPIが呼び出され処理が行われる
- クエリ文字列は特定したAPIのロジック内で処理されるパラメータ

### 判定:概ね正しい(「パス=API識別値」は一部正確化が必要)

### 実際の仕組み

**例**
```
GET /.well-known/device-bound-sessions?experiment_id=l2_enabled HTTP/1.1
```
- パス: `/.well-known/device-bound-sessions`
- クエリ文字列: `experiment_id=l2_enabled`(複数ある場合は`&`で連結)

**パスについての正確な理解**:
パスは「APIの識別値」に限らず、より広く「サーバ内の何らかの処理または資源を指し示す識別子」。大きく2パターンに分かれる。

1. **静的ファイルへのマッピング**: パスがサーバ上の実ファイル(画像・HTML等)に対応し、そのまま返される
2. **動的処理へのルーティング**: パス(+メソッド)に応じて、対応するプログラム(コントローラー)が実行される

**クエリ文字列**: サーバ側のプログラムコード内で取り出され(例: Flaskの`request.args.get('experiment_id')`)、条件分岐やDB検索条件などに使われる。あなたの認識通りで正確。

**全体の流れ**:
```
1. ブラウザ: GET /search?q=tokyo HTTP/1.1 を送信
2. Webサーバ: リクエストを受信
3. ルーティング: パス"/search"に対応する処理を探す
4. 該当処理が実行され、クエリパラメータ"q=tokyo"を取得
5. その値を使ってDB検索などを実行
6. 結果をレスポンスとして返す
```

**注意点**: クエリ文字列はURLの一部として見える形で送信されるため、機密情報を入れるのは避けるべき。POSTの場合パラメータは通常ボディに入り、リクエストラインには表示されない。

## 3. ルーティング/コントローラーの仕組み

### あなたの認識
- ルーター・ルーティングはネットワーク機器ではなく、バックエンドの構成要素(コントローラー)のこと
- API名とその処理先を対応付けたファイルを用意しておき、リクエスト内容から該当するAPIを探し出し、対応する処理を実行する、というのが現代の動的処理の仕組み

### 判定:正しい

### 実際の仕組み
フレームワーク(Flask, Django, Express, Spring等)が「パス ⇔ 対応する処理(関数)」の対応表(ルーティングテーブル)を持つ。

```python
# Flaskの例
@app.route('/users')
def get_users():
    # ユーザー一覧を返す処理
    ...

@app.route('/users/<id>')
def get_user(id):
    # 特定ユーザーを返す処理
    ...
```

**重要なポイント**: 処理は「パス」だけでなく「パス + メソッド」の組み合わせで決まる。
- `/users`へのGET → 一覧取得
- `/users`へのPOST → 新規作成

のように、同じパスでもメソッドによって呼ばれる処理(コントローラー)が異なるのが一般的な設計。

## 4. 静的ファイル配信とパストラバーサル

### あなたの認識
- 静的ファイルへのマッピングは、リクエストに対してサーバのリソース上にあるファイルを渡す際に利用される
- 渡すリソースの内容に制限を設けないことで攻撃される

### 判定:正しい(脆弱性名: パストラバーサル)

### 実際の仕組み
ユーザーから渡されたパスをそのままファイルシステムのパスに変換してファイルを読み込む実装の場合に発生する。

```python
# 脆弱な実装例
@app.route('/files/<filename>')
def get_file(filename):
    return open(f'/var/www/uploads/{filename}').read()
```

**通常のリクエスト**:
```
GET /files/photo.jpg HTTP/1.1
→ /var/www/uploads/photo.jpg を返す(意図通り)
```

**攻撃リクエスト**:
```
GET /files/../../../../etc/passwd HTTP/1.1
→ /var/www/uploads/../../../../etc/passwd
→ 結果的に /etc/passwd を返してしまう
```

`../`(1階層上に戻る)を繰り返すことで、本来アクセスできないファイル(設定ファイル、パスワードファイル、ソースコード等)を読み取られる可能性がある。

**対策**:
- 渡されたファイル名から`../`等の危険な文字列を除去・検証
- アクセス許可ディレクトリを限定し、そこから出られないようにする
- ファイル名を直接使わず、IDなどに変換してから内部でパスを解決

**Burpでの診断手法**: `../`を含むパラメータを試して、レスポンス内容(エラーかファイル内容が返るか)を確認する。

## 5. メソッド

### あなたの認識
- メソッドはリクエストでサーバ側に要求する内容であり、リクエストラインに記述される
- GET: ファイルやパラメータ等のリソースを取得
- POST: サーバ側に何らかの処理を要求する
- DELETE: リソースの削除
- PUT: リソースの更新
- PATCH: 部分的な更新

### 判定:概ね正しい(POSTの説明に補足必要)

### 実際の仕組み・補足

| メソッド | 説明 |
|---|---|
| GET | リソースの取得。認識通り |
| POST | 「サーバ側に何らかの処理を要求する」という認識でも大枠は正しいが、より正確には「新しいリソースの作成」や「データの送信(フォーム送信等)」が主目的。冪等性(同じリクエストを何度送っても結果が同じか)がない点がGETとの大きな違い |
| PUT | リソースの更新、または存在しない場合は作成。送ったデータでリソース全体を置き換えるイメージ。冪等性あり |
| PATCH | 部分的な更新。認識通り |
| DELETE | リソースの削除。認識通り |

### 今後学習すべき項目(言及していなかったメソッド)
- **HEAD**: GETと同じだがボディを返さない(ヘッダーだけ確認したい時に使う)
- **OPTIONS**: そのリソースがサポートするメソッドを確認する。CORSのプリフライトリクエストで使われる(7章のCORS参照)

## 6. ステータスコード

### あなたの認識
- ステータスコードはレスポンスのステータスラインに表示されるコード
- 100番台: 処理の実行中
- 200番台: 処理の成功
- 300番台: サービスサーバのプログラム処理へのリダイレクト
- 400番台: クライアント側のエラー
- 500番台: サーバ側のエラー

### 判定:300番台のみ修正が必要。修正後は正しい

### 実際の仕組み

| 番台 | 意味 |
|---|---|
| 1xx | 情報提供。処理が継続中であることを示す(例: 100 Continue) |
| 2xx | 成功 |
| 3xx | **クライアントに対して「別のURLにアクセスしてください」と指示するもの**。サーバ内部処理の話ではない |
| 4xx | クライアント側のエラー(例: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found) |
| 5xx | サーバ側のエラー(例: 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable) |

### 3xx(リダイレクト)の仕組み詳細
300番台のレスポンスを受け取ると、ブラウザはレスポンスヘッダーの`Location`に記載されたURLへ**自動的に次のリクエストを送信**する。

```
HTTP/1.1 302 Found
Location: https://example.com/login
```

このレスポンスを受け取ったブラウザは、自動的に`https://example.com/login`へGETリクエストを送る。

**主なコード**:
- **301 Moved Permanently**: リソースが永久に移動(検索エンジンはリンクを更新)
- **302 Found / 303 See Other**: 一時的なリダイレクト
- **304 Not Modified**: リソースに変更なし。キャッシュを使用(`Location`なし、ボディも返らない)

**Burpでの注意点**: 自動リダイレクト追従(Follow redirections)の設定により、302レスポンス自体(Locationヘッダー、Set-Cookieの内容)を見落とすことがある。診断時は自動追従を切って確認することが重要(認証バイパス等の発見につながる場合がある)。

## 7. ヘッダー・HTTP/1.1とHTTP/2の違い(今後学習すべき内容)

ここから先は、自身で詳しい言及をしていなかったため、説明内容をまとめたもの。今後の学習項目として整理。

### 7-1. Cookie / Set-Cookie 【重要度: 非常に高い】

- セッション管理・認証の基本
- レスポンスの`Set-Cookie`でサーバがクライアントに値を保存させる
- 以降のリクエストでブラウザが自動的に`Cookie`ヘッダーに付けて送る

```
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax
```

知っておくべき属性:
- `HttpOnly`: JavaScriptからアクセス不可(XSS対策)
- `Secure`: HTTPS通信でのみ送信
- `SameSite`: クロスサイトでCookieを送るかどうか(CSRF対策に関連。Strict/Lax/None)

### 7-2. Authorization 【重要度: 高い】

- リクエスト側のヘッダー。サーバに対して「自分は誰か」を証明する情報を送る
- 代表的な形式:
  - `Authorization: Basic <base64エンコードされたuser:pass>`
  - `Authorization: Bearer <トークン>`(JWTなど、APIで非常によく使われる)

```
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
```

Burpでの診断パターン: トークンを改ざんしたり別ユーザーのトークンに差し替えて、権限チェック(認可制御)の不備を確認する。

### 7-3. Content-Type 【重要度: 高い】

リクエスト・レスポンス両方で使われ、ボディのデータ形式を示す。

- `application/json`: JSON形式
- `application/x-www-form-urlencoded`: フォームデータ(`key=value&key2=value2`形式)
- `multipart/form-data`: ファイルアップロードを含むフォーム
- `text/html`: HTML

サーバはこのヘッダーを見てボディの解析方法を決めるため、偽装により予期しない動作を引き起こす攻撃手法もある。

### 7-4. CORS関連ヘッダー 【重要度: 高い(やや複雑)】

CORS(Cross-Origin Resource Sharing) = あるオリジン(サイト)のJavaScriptが、別オリジンのリソースにアクセスすることを許可するかどうかをブラウザが制御する仕組み。

主なヘッダー:
- `Origin`(リクエスト側): どのオリジンからのリクエストか
- `Access-Control-Allow-Origin`(レスポンス側): どのオリジンからのアクセスを許可するか
- `Access-Control-Allow-Credentials`: Cookie等の認証情報を含むリクエストを許可するか
- `Access-Control-Allow-Methods` / `Access-Control-Allow-Headers`: 許可するメソッド/ヘッダー

OPTIONSメソッド(プリフライトリクエスト)と密接に関わる。CORSの誤設定は実際の脆弱性として頻出。

### 7-5. ヘッダー学習の到達目安

1. **Cookie/Authorization**: 認証・セッション管理がどう行われているかを読み取れる。Burpで値を改ざんして挙動を確認する発想ができること
2. **Content-Type**: リクエスト/レスポンスのボディがどう解釈されているかを把握できること
3. **CORS**: なぜこのヘッダーが付いているのか、このサイトは他オリジンからのアクセスをどこまで許可しているかを読み取れること

### 7-6. HTTP/1.1とHTTP/2の違い

「データの表示形式(バイナリ vs テキスト)の違い」は本質ではなく、それは結果(副産物)。本質的な違いは以下。

**1. 多重化(Multiplexing) ※最重要**
- HTTP/1.1: 1つのTCP接続では基本的にリクエストを1つずつ順番に処理(レスポンスが返るまで次のリクエストが待たされる = Head-of-Line Blocking)。これを回避するため、ブラウザは複数のTCP接続を同時に張って並列化していた
- HTTP/2: 1つのTCP接続上で複数のリクエスト/レスポンスを**同時に・並行して**送受信できる(フレームに分割して多重化)。接続のオーバーヘッドが減りパフォーマンス向上

**2. ヘッダー圧縮(HPACK)**
- HTTP/1.1: 毎回ヘッダー(Cookie等含む)をテキストでそのまま送る
- HTTP/2: HPACKでヘッダーを圧縮し、繰り返し送られるヘッダー情報の冗長性を削減

**3. サーバープッシュ(廃止予定/廃止済み)**
- HTTP/2で導入されたが、ブラウザサポートが進まず実質使われていない(知識として軽く触れる程度)

**4. 暗号化との関係**
- HTTP/2は仕様上は平文(h2c)も可能だが、主要ブラウザはHTTPS(TLS)上でのみHTTP/2を使う実装。実用上は「HTTP/2 = 実質HTTPS」という認識で近い

**5. リクエストラインの構造変化**
- HTTP/2には「リクエストライン」という行自体が存在しない
- HTTP/1.1の`GET /path HTTP/1.1`は、HTTP/2では`:method`, `:path`, `:scheme`, `:authority`という擬似ヘッダー(pseudo-header)に分解される
- ヘッダー情報→HEADERSフレーム(HPACK圧縮されたバイナリ)、ボディ→DATAフレーム という形でバイナリのフレーム単位に分割して送信される
- Burpが`GET /path HTTP/2`のように表示するのは、本来バラバラな擬似ヘッダーやフレーム構造を、ツールがHTTP/1.1風の見慣れた形式に**再構成して表示**しているため(ワイヤー上の実データとは別物)

**Burpを使う上での実用的な意味**:
- HTTP/1.1ではリクエストを1つずつ送る感覚で見られるが、HTTP/2では並行して複数のリクエストが飛んでいるのを「HTTP history」で見ることになる
- Repeaterで手動リクエストを送る際は基本的に1リクエスト/1レスポンスとして扱えるため、診断作業自体への影響は限定的だが、「なぜこのページで多数のリクエストが同時に発生しているのか」を理解する際にこの違いが関係する
